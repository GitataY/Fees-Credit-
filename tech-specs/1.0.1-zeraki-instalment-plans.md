**Source feature spec**: features/1.0.1-zeraki-instalment-plans.md
**Last updated**: 2026-04-01

## Overview

**What this implements**: A structured instalment plan layer on top of Zeraki Finance's existing fee records — allowing parents to request payment plans via app or SMS, bursars to approve/adjust/decline via a queue, and the system to track per-instalment adherence and send automated SMS at every state transition.

**Key technical decisions**:
1. **Payment allocation is synchronous** — FIFO allocation runs inline within the webhook/manual-entry handler, not as a background job, to ensure immediate SMS dispatch and dashboard consistency
2. **Adherence metrics are stored, not computed on read** — `on_time_rate` and `completion_rate` are written to the plan record after every allocation event (per BR-12); no aggregation queries at read time
3. **Duplicate detection gates payment application** — when a duplicate flag is raised, neither payment is applied until the bursar resolves it; the M-Pesa webhook payment is held in `PENDING_REVIEW` status
4. **SMS sessions use a dedicated sessions table** — inbound PLAN/BAL keyword sessions are tracked server-side with TTL; no client session state required
5. **Scheduled jobs are idempotent** — reminder and missed-detection jobs use processed flags on the instalment record to prevent double-firing on retries

**Dependencies**:
- Existing Zeraki Finance: school records, student enrolment, fee balance records, M-Pesa paybill webhook, bursar auth, outbound SMS infrastructure
- New infrastructure: inbound SMS routing (OQ-06 — not yet confirmed)
- New infrastructure: parent-facing app (OQ-05 — not yet confirmed)
- No new external payment integrations — all payments flow through existing paybill

### Inbound SMS — Keyword Handler

#### POST /api/sms/inbound
**Purpose**: Handle inbound SMS keywords (PLAN, BAL) from parents
**Actor**: SMS provider webhook
**Auth**: Shared secret header (`X-SMS-Provider-Secret`) — provider-specific

**Request** (provider-normalised payload):
```json
{
  "from": "+254700000000",
  "to": "+254800000000",
  "body": "PLAN",
  "received_at": "2026-04-01T09:00:00Z"
}
```

**Response**: `200 OK` (provider acknowledgement — content irrelevant)

**Handler logic** (internal — not an API response):
1. Look up parent by `from` phone number across all opted-in schools
2. Route to active SMS session if one exists for this number; else create new session
3. Dispatch to `PLAN` or `BAL` keyword handler
4. Respond via outbound SMS using school's sender ID

**Session schema** (see `sms_sessions` table below):
- PLAN session: 10-minute TTL; stores `school_id`, `student_id` (if resolved), `step` (template_selection | confirmation), `template_id`
- BAL: stateless — no session required; respond immediately

**Notes**: OQ-06 (inbound SMS infrastructure) must be resolved before this endpoint can be built. The phone number → school routing assumes each parent phone number maps to exactly one opted-in school. Multi-school parents (OQ-08) need disambiguation logic.

### Scheduled Jobs (System — no HTTP actor)

#### Job: instalment-reminder
**Schedule**: Daily, configurable time (recommend 08:00 school local time)
**Purpose**: Send reminder SMS for instalments due within the reminder window (OQ-01 — window TBD)

**Logic**:
1. Find all PENDING instalments where `due_date` is within `[reminder_lead_days]` days from today
2. Filter: plan status = ACTIVE, `reminder_sent_at` IS NULL (idempotency guard)
3. For each: calculate `remaining_amount = instalment.amount_cents - instalment.paid_amount_cents`
4. Send reminder SMS using `remaining_amount` (not `instalment.amount_cents`) — per journey OQ (partial payment case)
5. Set `reminder_sent_at = now()`, `reminder_delivery_status` per provider response

**Idempotency**: `reminder_sent_at IS NULL` guard prevents double-send on job retry.

---

#### Job: instalment-missed-detection
**Schedule**: Daily, runs after paybill settlement window (recommend 23:00)
**Purpose**: Mark overdue instalments as MISSED, notify parent and bursar (OQ-02 — grace period TBD)

**Logic**:
1. Find all PENDING instalments where `due_date + [grace_period_days] < today`
2. For each: mark instalment MISSED, recalculate `on_time_rate` and `completion_rate` on plan
3. Send overdue SMS to parent; send alert SMS to bursar
4. If all remaining instalments on the plan are now MISSED and term has ended: transition plan to DEFAULTED
5. Mark processed with `missed_processed_at = now()` (idempotency guard)

**Idempotency**: Check `status != 'missed'` before processing — prevents double-notification on retry.

## Integration Points

### M-Pesa Paybill Webhook (existing)
**What it does**: Delivers payment events from M-Pesa to Zeraki Finance when a parent pays fees
**Modification required**: Extend existing webhook handler to check for ACTIVE instalment plans before processing. If a plan exists: run FIFO allocation logic, update adherence metrics, send plan SMS. If no plan: existing fee receipting behaviour unchanged.
**Failure mode**: Webhook delivery failure or handler error
**Fallback**: M-Pesa webhook providers typically retry on non-2xx response. Handler must be idempotent (check if payment already allocated before re-allocating). Recommend idempotency key based on M-Pesa transaction reference.

> **Architectural requirement — webhook SMS must be async**: M-Pesa C2B callbacks have a short timeout window (typically 5–10 seconds). If the webhook handler blocks on FIFO allocation + adherence recalculation + synchronous SMS send and exceeds that window, the M-Pesa provider will retry the callback — creating a duplicate payment event. The idempotency guard will catch it, but the cleaner fix is to split the handler into two stages:
> 1. **Synchronous** (within the callback window): FIFO allocation, adherence metric recalculation, duplicate flag check, persist all DB writes. Return `200` to M-Pesa immediately.
> 2. **Async** (fire-and-forget to a fast internal queue): SMS dispatch (payment received, plan complete).
>
> This applies **specifically to the M-Pesa webhook path**. Bursar manual payment (`POST /api/instalment-plans/:plan_id/payments`), plan approval, and all other bursar-initiated flows keep synchronous SMS — they do not have a provider timeout constraint and the `sms_sent: true` response field is meaningful to the UI.

### Outbound SMS (existing)
**What it does**: Sends all parent and bursar SMS notifications via the school's registered sender ID (BR-14)
**Events triggering SMS**: Plan request received, plan active, plan active (adjusted), plan declined, instalment reminder, payment received, plan complete, overdue notice, bursar missed alert, plan cancelled, BAL response, PLAN session messages
**New requirement**: All plan-related SMS must be sent synchronously within the request/job that triggers them (`sms_sent: true` in API response) — not deferred to a background queue — to ensure immediate parent notification.
**Failure mode**: SMS provider failure or delivery failure
**Fallback**: Log `reminder_delivery_status = 'failed'` on instalment record. Bursar dashboard surfaces failed reminders. Bursar can manually resend. For non-reminder SMS (plan active, payment received), log failure and surface in bursar audit log — no automatic retry to avoid double-SMS on retry.

### Inbound SMS (new — OQ-06 dependent)
**What it does**: Routes inbound PLAN and BAL keyword messages from parents to the Zeraki Finance SMS handler
**Requirement**: A dedicated inbound SMS number (short code or long number) that Zeraki controls and can receive messages on, routable to POST /api/sms/inbound
**Failure mode**: Provider connectivity failure — inbound messages lost
**Fallback**: No fallback for lost inbound messages. Provider SLA is the mitigation. Ensure provider supports delivery receipts.
**Blocker**: This integration cannot be built until OQ-06 is resolved (shared vs. per-school number, provider choice).

### Parent App (new — OQ-05 dependent)
**What it does**: Delivers Path A (app-initiated plan requests) and exposes the fee screen with offer states
**API surface**: GET /api/students/:student_id/fee-summary, GET /api/schools/:school_id/plan-templates/:template_id/preview, POST /api/plan-requests
**Auth**: Parent-scoped JWT. Auth mechanism (how parents log into the Zeraki app) is not specified — flag for the app team.
**Blocker**: Cannot be built until OQ-05 is resolved (existing app vs. new build).

## Security Considerations

- **Authorization per school**: Every bursar-facing endpoint must verify the requesting user belongs to the school in the path parameter. A bursar at School A must not access School B's plans. Implement as middleware that validates `school_id` in the path against the authenticated user's school membership.
- **Parent-scoped tokens**: Parent app endpoints (fee summary, template preview, plan request) must use student-scoped tokens — a parent may only access their own students' data. Validate `student_id` against the token's associated students.
- **Idempotency for plan creation**: `POST /api/plan-requests` and `POST /api/instalment-plans` are financial-record-creating endpoints. Accept a client-supplied `idempotency_key` header; return the original response on duplicate keys within a 24-hour window.
- **Input validation — payment dates**: `payment_date` in manual payment entries cannot be in the future (validated server-side, not just client-side). Backdating to prior terms is an OQ (OQ-13) — default to blocking prior-term backdating until resolved.
- **Duplicate flag resolution audit**: Every `POST /api/duplicate-flags/:flag_id/resolve` call logs `resolved_by` and `resolved_at`. This is a financial audit trail — do not allow deletion of flag records.
- **SMS inbound auth**: The inbound SMS webhook (`POST /api/sms/inbound`) must authenticate using a shared secret header from the provider. Reject requests without the header with `401`.
- **PII handling**: `parent_phone` is stored on plan request and plan records. Ensure it is excluded from any logging middleware. Phone numbers in SMS session records should be treated as PII.
- **Rate limiting**: `POST /api/plan-requests` (parent app) — limit to 5 requests per student per hour to prevent accidental duplicate submissions. `POST /api/sms/inbound` — rate limit per `from` number to prevent SMS flooding.

## Performance Considerations

- **Dashboard query performance**: The dashboard endpoint aggregates across all plans, instalments, and payments for a school's term. At 150 active plans with 4 instalments each = 600 instalment rows per school per term — manageable without caching for MVP. Add a materialized view or Redis cache if query time exceeds 500ms at load.
- **Adherence metrics stored, not computed**: `on_time_rate` and `completion_rate` are written to `instalment_plans` on every allocation event (BR-12). This means the dashboard read requires no aggregation — it reads stored values directly. Protect this by ensuring no code path modifies instalment status without also recalculating and persisting the metrics.
- **FIFO allocation query**: The allocation job queries `SELECT * FROM instalments WHERE plan_id = ? AND status = 'pending' ORDER BY sequence ASC LIMIT 1`. The `(plan_id, status)` index makes this fast. Keep the allocation logic in a single transaction.
- **Scheduled jobs — isolation**: Reminder and missed-detection jobs must not lock each other. Use row-level locking (`SELECT ... FOR UPDATE SKIP LOCKED`) when marking instalments as processed to allow parallel processing of large instalment sets.
- **SMS delivery — synchronous for bursar flows, async for webhook**: Outbound SMS is called synchronously within bursar-initiated request handlers (plan approval, manual payment, etc.) — `sms_sent: true` in the response is meaningful to the UI. The M-Pesa webhook path is a hard exception: SMS must be dispatched async after the DB writes commit (see Integration Points). If the SMS provider SLA exceeds 2s for bursar flows, revisit. OTQ-07 covers the provider SLA decision.
- **Race condition — unique partial index must be caught, not swallowed**: The `UNIQUE (student_id, term_id) WHERE status = 'active'` partial index on `instalment_plans` is the authoritative guard against duplicate active plans (BR-01). A real race exists in normal school-hour usage: a parent sends `PLAN` via SMS at the same moment a bursar creates a plan via Path C. Both requests pass the application-layer duplicate check before either commits — the second DB write throws a unique constraint violation. **Every handler that inserts into `instalment_plans` must explicitly catch this constraint error and return `409 PLAN_ALREADY_EXISTS`**, not a `500`. This includes the SMS keyword handler, the parent app request handler, and the bursar direct-creation handler. Treat this as a required error path, not an edge case.
