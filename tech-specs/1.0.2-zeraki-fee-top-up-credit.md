**Source UI spec**: ui-specs/1.0.2-zeraki-fee-top-up-credit.md

> **Note**: No BDD spec exists yet. Error codes are derived from the feature spec's error contracts. Revisit when `specs/1.0.2-zeraki-fee-top-up-credit.md` is produced.
>
> **Read alongside**: `tech-specs/1.0.1-zeraki-instalment-plans.md`. This spec extends the M-Pesa webhook handler defined there and inherits its SMS async pattern. Patterns documented in 1.0.1 are referenced here rather than repeated.

---

## Overview

**What this implements**: An embedded school fee credit product — eligibility assessment, offer delivery, two-step consent (app and SMS), disbursement coordination with a lending partner, waterfall repayment allocation built into the existing M-Pesa paybill webhook, and a read-only bursar dashboard.

**Key technical decisions**:
1. **Disbursement is async webhook-to-webhook** — Zeraki sends a disbursement request to the lending partner API and waits for a confirmation webhook; there is no synchronous disbursement response. The parent app bridges the ACCEPTED → DISBURSED gap with a processing state (resolved by OQ-UI-05 — see Performance section).
2. **Waterfall allocation extends the 1.0.1 webhook handler** — same async pattern: return `200` to M-Pesa immediately, run waterfall allocation + 1.0.1 FIFO (if Layer 1 plan also active), then dispatch SMS async. Credit repayment share takes priority over instalment plan allocation (BR-08).
3. **Consent version is stored, not inferred** — the exact consent disclosure text shown to the parent at each step is versioned and stored on the arrangement record (BR-09). Consent text changes require a new version; existing arrangements are immutable.
4. **STOP is session-scoped, not carrier-level** — the application layer must intercept STOP during a credit consent SMS session before it reaches any carrier opt-out processing (BR-20). This is a hard constraint on the SMS provider choice.
5. **Eligibility batch is a scheduled, idempotent computation job** — runs at term start (and optionally mid-term per OQ-03); stores results in `eligibility_records`; is safe to re-run without duplicating offers.
6. **B2B repayment transfer is queued, not per-transaction** — repayment shares are accumulated in `b2b_transfer_batches` and transferred to the lending partner on a schedule (OQ-08 — daily recommended pending partner confirmation).

**Dependencies**:
- `tech-specs/1.0.1-zeraki-instalment-plans.md` — M-Pesa webhook handler, inbound SMS infrastructure (OQ-06), Layer 1 plan adherence data for Path 2 eligibility
- Lending partner integration — API contract not yet confirmed (OQ-01 / Pezesha unvalidated). This spec defines the interface Zeraki needs; the partner must confirm field names, auth, and error codes before implementation.
- Consent text legal review — OQ-13 (CBK + KDPA). Consent version 1.0 cannot go live without legal sign-off.
- Parent-facing app — OQ-05 (same blocker as 1.0.1 Screens 2–5)

---

## Stack Assumption

Same as 1.0.1: spec is stack-agnostic. Assumes SQL (PostgreSQL). The eligibility scoring computation (five weighted components per parent per term) is CPU-bound — flag if the existing backend is single-threaded or has tight request timeouts; the batch job should run outside the web request cycle.

### Lending Partner Webhooks

#### POST /webhooks/lending-partner/disbursement-confirmation
**Purpose**: Receive disbursement confirmation (or failure) from the lending partner after a disbursement request
**Actor**: Lending partner (webhook push)
**Auth**: HMAC signature on request body using shared secret (`X-Partner-Signature` header) — exact scheme to be confirmed with partner

**Request** (Zeraki-normalised from partner payload — see Data Contracts):
```json
{
  "zeraki_arrangement_id": "uuid",
  "partner_reference": "PZS-TXN-98765",
  "disbursed_amount_cents": 450000,
  "disbursed_at": "2026-04-01T09:12:00Z",
  "status": "success | failed",
  "failure_reason": "string | null"
}
```

**Handler logic (success)**:
1. Validate HMAC signature — reject `401` if invalid
2. Look up arrangement by `zeraki_arrangement_id`; verify status is ACCEPTED — reject `409` if already processed (idempotency)
3. Transition arrangement: ACCEPTED → DISBURSED → ACTIVE_REPAYMENT (single transaction)
4. Record `disbursed_amount_cents`, `partner_reference`, `disbursed_at`
5. Collect Zeraki fee (BR-11 — mechanism TBD per OQ-12)
6. Update `school_credit_term_summary.total_disbursed_cents` (increment)
7. Dispatch async: disbursement confirmation SMS to parent

**Handler logic (failure)**:
1. Validate signature
2. Transition arrangement: ACCEPTED → DISBURSEMENT_FAILED
3. Record `failure_reason` (stored in `disbursement_failure_reason` — for ops and audit only)
4. Dispatch async: disbursement failed SMS to parent

> **`disbursement_failure_reason` handling rule**: The lending partner may send technical error strings such as `INSUFFICIENT_FUNDS_IN_POOL`, `KYC_LIMIT_EXCEEDED`, or similar. These strings must never surface to the parent or the bursar. `disbursement_failure_reason` is stored exclusively for Zeraki ops and audit trail — it must not be returned in any API response visible to the parent app or the bursar dashboard. The parent-facing SMS always uses the generic failure copy (defined in Outbound SMS section) regardless of the underlying reason. The bursar dashboard always shows "Disbursement failed — contact Zeraki support." No branching on specific failure codes in the parent or bursar UI.

**Response**: `200 OK` (always, after signature validation — partner must not retry on business logic errors)

### Inbound SMS — FEES Keyword Extension

The existing `POST /api/sms/inbound` handler (defined in 1.0.1 tech spec) is extended to handle the `FEES` keyword for credit consent. The session flow is multi-step:

| Step | Inbound from parent | System response | Session state |
|------|--------------------|-----------------|----|
| 1 | `FEES` | Consent Step 1 SMS | `credit_consent_step_1` |
| 2 | `YES` | Consent Step 2 SMS | `credit_consent_step_2` |
| 3 | `CONFIRM` | Acceptance confirmation SMS; record consent; trigger disbursement | terminal |
| Any | `NO` or `STOP` | Session terminated (see BR-20 note) | terminal |
| Any | Invalid reply | "Reply YES to continue or NO to cancel." | unchanged |

**Session TTL**: 30 minutes (vs. 10 minutes for PLAN keyword — feature spec Flow 5).

**BR-20 — STOP interception**: When `STOP` is received during a credit consent session, the application must:
1. Look up the active session for this phone number
2. If an active credit consent session exists: terminate that session only; do NOT forward STOP to the carrier opt-out layer; reply (optionally): no reply, or "Your fee top-up request has been cancelled."
3. If no active credit consent session: STOP should be handled per normal carrier opt-out rules

**Implementation constraint**: This requires the SMS provider to support application-layer keyword interception before carrier-level STOP processing. Must be confirmed with the chosen provider before implementation. If the provider cannot support this, a different keyword (e.g. `CANCEL`) must be used for session termination in credit flows — `STOP` must remain the carrier opt-out word. **This is a go/no-go technical requirement for the SMS provider selection.**

### Scheduled Jobs (System)

#### Job: eligibility-assessment
**Schedule**: Start of each term for all opted-in schools; optionally mid-term (OQ-03)
**Purpose**: Compute credit scores and create eligibility records for qualifying parents; send offers

**Logic** (per school per term):
1. Identify all students with outstanding fee balance this term at this school
2. For each student's parent, assess Path 1 (payment speed, requires 4+ terms) and Path 2 (Layer 1 on_time_rate ≥ 0.75 on completed plan)
3. For each parent who meets at least one path: compute five-component credit score (see Credit Scoring Model in feature spec)
4. Map score to tier (A/B/C); calculate maximum offer amount; exclude if below KES 3,000 floor
5. Check: does this parent already have an offer or active arrangement this term? If yes, skip (idempotency guard)
6. Create `eligibility_record` for qualifying parents
7. Send eligibility records to lending partner for pre-approval (batch API call or per-parent — confirm with partner)
8. On partner pre-approval response: store `partner_approved_amount`; create `credit_arrangement` in OFFERED status
9. Dispatch offer notification (push notification + in-app banner for app path; offer SMS for SMS path)

**Idempotency**: Check for existing `eligibility_record` for this `(student_id, term_id)` before computing. Safe to re-run — skips already-assessed parents.

**Performance note**: Credit score computation is CPU-bound. At 4,000 schools × ~50 eligible parents = up to 200,000 score computations per term-start batch. This must run as a background job, not in a web request. Recommend a job queue (e.g. Bull, Sidekiq, or equivalent) with concurrency tuned to available CPU. Batch jobs per school to limit memory footprint.

---

#### Job: offer-expiry
**Schedule**: Daily
**Purpose**: Expire OFFERED arrangements that have passed `offer_expires_at` without response

**Logic**:
1. Find all `credit_arrangements` where `status = 'offered'` and `offer_expires_at < NOW()`
2. Transition to EXPIRED
3. Send expiry SMS to parent (using updated copy from UI spec upstream feedback: "Eligible parents will be reassessed at the start of next term.")
4. Mark processed with `expired_at = NOW()` (idempotency guard)

---

#### Job: b2b-transfer
**Schedule**: Daily (recommended — OQ-08 pending partner confirmation)
**Purpose**: Transfer accumulated repayment shares to the lending partner

**Logic**:
1. Find all `waterfall_allocations` where `b2b_transfer_status = 'queued'`
2. Group by `lending_partner_id`; sum `repayment_share_cents` per partner
3. Execute B2B transfer via lending partner's API or MPESA B2B
4. Update all included allocations: `b2b_transfer_status = 'sent'`, `b2b_transfer_reference = <ref>`
5. On failure: mark as `b2b_transfer_status = 'failed'`; surface in ops dashboard for manual retry

## Integration Points

### Lending Partner API (Pezesha — unvalidated, OQ-01)
**What it does**: Receives disbursement requests from Zeraki; disburses directly to school paybill; sends disbursement confirmation webhook; receives B2B repayment transfers; receives AT_RISK notifications
**API/SDK**: REST. Exact auth (OAuth2, API key, mutual TLS), base URL, and field names are not yet confirmed — Pezesha must provide an integration guide. This spec defines the Zeraki-side interface contract; the partner contract section is a placeholder until OQ-01 is resolved.
**Failure mode — disbursement request**: Lending partner API unreachable or returns rejection
- Unreachable: retry with exponential backoff per OQ-07 (retry count and interval TBD). After max retries, flag arrangement for manual review. Do not transition to DISBURSEMENT_FAILED until max retries exhausted.
- Rejection: transition immediately to DISBURSEMENT_FAILED. Send parent SMS. Record `failure_reason`.
**Failure mode — B2B transfer**: Transfer fails; `b2b_transfer_status = 'failed'`; surface in ops dashboard for manual retry. Lending partner must confirm whether failed transfers affect the repayment record on their side.
**Fallback**: No automated fallback. Manual investigation path required (OQ-06).
**Blocker**: Nothing in Phase 2+ can be fully tested without a lending partner sandbox environment.

### M-Pesa Paybill Webhook (extension of 1.0.1)
**What it does**: The existing webhook handler (defined in 1.0.1 tech spec) is extended to run waterfall allocation before Layer 1 FIFO allocation
**Extended handler order**:
1. Return `200` to M-Pesa immediately (same async-first pattern as 1.0.1)
2. Check for ACTIVE_REPAYMENT credit arrangement for this student
3. If found: calculate repayment share; update `credit_arrangements.remaining_balance_cents`; write `waterfall_allocation` record; queue B2B transfer; update `school_credit_term_summary`; check for completion
4. Take `payment.amount - repayment_share` as the remainder; pass remainder to Layer 1 FIFO allocation handler (1.0.1 logic — unchanged)
5. Async: dispatch waterfall payment SMS (or completion SMS if arrangement completed); dispatch 1.0.1 payment SMS
**Idempotency**: Check `waterfall_allocations` for existing record with this `payment_id` before processing — same guard as 1.0.1.

> **Manual payment waterfall requirement**: The waterfall check does not apply only to M-Pesa webhook payments. If a parent has an ACTIVE_REPAYMENT credit arrangement and pays cash, the bursar records it via the Layer 1 manual payment endpoint (`POST /api/instalment-plans/:plan_id/payments`). That handler must also call the waterfall logic before running Layer 1 FIFO allocation — same priority order: credit repayment share first, Layer 1 plan second. The Layer 1 manual payment handler must be extended to check for an active credit arrangement on the student before proceeding with FIFO allocation. If an arrangement is found: calculate repayment share, write `waterfall_allocation` record, queue B2B transfer, update `school_credit_term_summary`, check for completion — then pass the remainder to FIFO allocation. The waterfall logic is the same code path regardless of payment origin (M-Pesa or manual); only the triggering entry point differs.

### Outbound SMS (extension of 1.0.1)
**New SMS events**: Credit offer (push), Consent Step 1, Consent Step 2, Offer accepted/disbursed, Offer expired, Waterfall payment applied, Credit completed, Disbursement failed, AT_RISK notification
**Updated expiry SMS copy** (from UI spec upstream feedback): `"[School]: Your fee top-up offer for [Student] has expired. Eligible parents will be reassessed at the start of next term."` — replaces the feature spec's copy ("Contact the school if you still need support").
**Loan language prohibition** (BR-14): All outbound SMS events must be reviewed against the prohibited language list before the template is committed. "Loan" must not appear in any parent-facing SMS. All templates must pass a keyword check at deploy time.

### Inbound SMS (extension of 1.0.1)
**STOP interception requirement** (BR-20): The SMS provider must support application-layer keyword capture before carrier-level STOP processing. This is a hard provider selection criterion. Confirm before signing a provider contract. If the provider cannot support this, `STOP` cannot be used as the session cancellation keyword — use `CANCEL` instead for credit flows.

## Security Considerations

- **Consent amount validation**: `POST /api/credit-arrangements/:arrangement_id/consent/step-2` validates `confirmed_credit_amount_cents` and `confirmed_total_repayment_cents` against the server-side arrangement record. This prevents a client-side manipulation where a modified app submits a confirmation for a different amount than what was displayed.
- **Consent version pinning**: The `consent_version` field is returned from the server at offer time and echoed back at each consent step. If the active consent version changes between offer and confirmation (e.g., a legal update mid-session), the server returns `CONSENT_VERSION_MISMATCH (422)` — the client must re-fetch and show the current version before the parent can confirm.
- **Lending partner webhook authentication**: `POST /webhooks/lending-partner/disbursement-confirmation` must verify an HMAC signature on the request body before processing. The exact scheme (SHA-256, SHA-512, header name) must be confirmed with the partner. Reject `401` on missing or invalid signature — do not process unauthenticated disbursement events.
- **Borrower reference hashing**: The lending partner disbursement request sends a hashed borrower reference, not raw PII (BR-10). The hash must be deterministic (same parent → same hash) but one-way. Recommend HMAC-SHA256 with a Zeraki-internal secret — this allows Zeraki to look up the parent from a partner reference without exposing PII in logs.
- **Eligibility data access**: `GET /api/students/:student_id/credit-offer` returns offer details scoped to the requesting parent's students. A parent must not be able to query another parent's eligibility or arrangement status.
- **Consent audit trail**: `consent_given_at`, `consent_channel`, `consent_version`, `confirmed_credit_amount_cents`, and `confirmed_total_repayment_cents` are immutable once written. No UPDATE path exists for these fields — they are write-once records. Implement at the application layer; consider a DB trigger as a secondary guard.
- **PII in SMS logs**: Arrangement IDs and student names in SMS templates are PII-adjacent. SMS outbound logs must not persist message body content beyond what is required for delivery status tracking. Confirm with the SMS provider what their data retention policy is for message bodies.
- **Rate limiting**: Consent Step 2 (`POST .../consent/step-2`) — limit to 3 attempts per arrangement. After 3 failed attempts (e.g., amount mismatch), lock the arrangement and require the parent to re-initiate via the offer screen. Prevents brute-force confirmation attempts on intercepted arrangement IDs.

## Performance Considerations

- **`DisbursedMetricHero` must never be a query-time SUM**: At 1,000 schools with 50 arrangements each, `SELECT SUM(disbursed_amount_cents) FROM credit_arrangements WHERE school_id = ? AND term_id = ?` is trivial. At 4,000 schools with 200 arrangements, it is still fast. However, the `school_credit_term_summary` pre-aggregated table eliminates this concern entirely and costs one extra write per status transition — use it from day one.
- **Eligibility batch parallelism**: The eligibility assessment job processes one school at a time in a queue. At 4,000 schools, even with 5-minute processing per school, serial execution takes 13+ days. Batch jobs must be parallelised across schools — each school's assessment is an independent unit of work. Recommend job queue with concurrency = available CPU cores × 0.75.
- **Credit score computation caching**: Payment history data (the input to the scoring model) does not change during the batch run. Cache the raw payment history for each parent at the start of the job (in-memory, per school batch) to avoid redundant DB reads when a parent has multiple students at the same school.
- **Screen 6 polling**: The parent app polls `GET /api/students/:student_id/credit-offer` every 5 seconds for up to 2 minutes post-consent to detect disbursement confirmation. At MVP load (say 100 concurrent consent acceptances at term start), this is 1,200 requests/minute for 2 minutes — manageable. If disbursement confirmation latency is consistently >60s (depends on lending partner), consider a push notification from the lending partner webhook instead of polling. Revisit after measuring real lending partner response times.
- **Waterfall allocation query**: Check for ACTIVE_REPAYMENT arrangement: `SELECT * FROM credit_arrangements WHERE student_id = ? AND status = 'active_repayment' AND term_id = ?`. The `(student_id, term_id, status)` index makes this a single-row lookup — fast.
