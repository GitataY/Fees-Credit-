## J1 — School Enables Instalment Plans

**Actor**: Headteacher / School Admin
**Mental state entering**: Cautiously curious. Has heard about the feature and wants to give parents a structured way to pay, but is concerned about whether turning this on means the school loses control — or worse, is committed to something it can't enforce.

### Trigger
Admin navigates to Settings → Fee Collection → Instalment Plans, either proactively or after a prompt from a Zeraki customer success contact.

### Journey Steps

**1. Landing on the settings screen**
Admin sees the feature description. The framing is important: this is a tool that gives parents dignity, not a collections mechanism. Admin is reading carefully — they want to understand what "enabling" actually means.

**2. Reading the commitment statement**
*"By enabling instalment plans, this school agrees that any student with an active plan that is being honoured will not be sent home for fee arrears during the plan period."*

This is the heaviest moment in J1. The admin pauses. Two things are happening simultaneously:
- Recognition: *"This is what we should already be doing."*
- Uncertainty: *"What does 'honoured' mean? What if we need to enforce?"*

The product's job here is to remove doubt without minimising the commitment. The statement should be clearly visible, not buried in small print. The **Enable Instalment Plans** button should not be the only element on screen — the commitment needs enough space to be read, not scrolled past.

> **Friction point — commitment anxiety**: Admin may want to show the commitment to their board or deputy before clicking. There is no share/copy mechanism noted in the feature spec. Consider whether a "Print / Save this page" affordance is worth adding, or whether the confirmation email (post-activation) should include the commitment text.

**3. Enabling the feature**
Admin clicks **Enable Instalment Plans**. System confirms and immediately prompts to set up templates. This transition is well-sequenced: enabling without templates would leave the feature half-live. Forcing templates before save is correct — but the prompt should feel like a natural next step ("Great — now let's set up the plan options parents will see") rather than a blocker.

**4. Creating plan templates (1–3)**
Admin creates templates. This is a configuration task, not a form-filling exercise — the admin is thinking about their school's financial calendar, not a product interface. Each template has a name and a schedule structure (e.g., "Weekly: 4 payments").

Key tension: admins will want to know *how* the amounts are calculated before they commit to publishing a template. The spec says amounts are calculated at request time from the student's outstanding balance. This needs to be surfaced during template creation: *"Amounts are calculated at the time a parent requests a plan, based on their child's outstanding balance."* Without this, admins will wonder if they're setting a fixed amount.

> **Friction point — template amount confusion**: Admin may think templates define a payment amount. The product must make clear that templates define structure only (count + interval), and amounts are calculated per-student at request time.

**5. Selecting approval mode**
Manual approval (default) vs. auto-approve. Most admins will leave this as manual for the first term, then switch to auto-approve once they trust the system. The default is correct. The mode choice should feel low-stakes to reverse.

**6. Saving settings**
System confirms: *"Instalment plans are now enabled. Parents can request plans via the Zeraki app or by SMS. Bursars can create plans directly from the student fee screen."*

The confirmation message orients the admin to what happens next — three channels are now live. This is the right level of specificity.

### After
The school is opted in. The admin's world is now: parents can request plans without calling the school. The bursar has a queue to manage. The admin will not interact with the feature again unless they need to add or deactivate templates, change the approval mode, or disable the feature for a new term.

### Established Patterns Used
- Irreversible action pattern: commitment statement + explicit confirmation button (adapted from friction resolution patterns — commitment, not deletion)
- System-prompted next step: template setup is surfaced immediately after activation

## J2 — Parent Requests a Plan via the Zeraki App (Path A)

**Actor**: Parent
**Mental state entering**: Relieved there's a digital option, but slightly anxious. Fee conversations are loaded. The parent has a real balance they can't pay all at once and wants to make a legitimate arrangement without the embarrassment of a face-to-face negotiation.

> **Note**: This path depends on OQ-05 (whether the Zeraki parent app exists or is a new build). The journey assumes the app exists and the parent is already authenticated.

### Trigger
Parent opens the Zeraki app, navigates to their child's fee screen, and sees a fee balance with an outstanding amount. They notice the **Request a Payment Plan** option.

### Journey Steps

**1. Arriving on the fee screen**
The outstanding balance is visible. The **Request a Payment Plan** CTA needs to be clearly discoverable but not alarming — it should read as a helpful option, not a crisis intervention. Positioning matters: next to or below the balance figure, not in a warning state.

**2. Selecting a template**
Parent sees the school's available plan options (up to 3). Template names like "Weekly: 4 payments" are functional but dry. The parent wants to understand what they're agreeing to before they commit.

> **Friction point — abstract template names**: "Weekly: 4 payments" doesn't tell a parent anything about amounts until the next screen. Consider whether templates should show a preview calculation on this screen ("approx. KES [amount] / payment") or whether this is deferred to the schedule preview step.

**3. Reviewing the proposed schedule**
System calculates and displays the instalment schedule: due dates and KES amounts per instalment. This is the moment of truth. The parent either nods ("yes, that's manageable") or reconsiders ("that's more than I thought").

This screen carries the most UX weight in J2. The parent needs to see:
- Total plan amount
- Number of instalments
- Each due date and amount
- Where to pay (the paybill — same as always)

The "pay to paybill as usual" message should appear here, not just in the SMS. Parents who don't read SMS carefully need to know nothing about their payment channel changes.

**4. Confirming**
Parent taps **Confirm**. System creates a PENDING_APPROVAL request.

**5. Receiving SMS confirmation**
Parent receives: *"Your plan request for [Student name] is being reviewed. You'll hear back shortly."*

The gap between confirmation and hearing back is an unknown. If the bursar takes 3 days, the parent may call the school anyway — defeating the purpose. The product cannot control bursar response time, but the journey should surface the expected review window if one exists.

> **Friction point — waiting anxiety**: Parent does not know how long approval takes. No SLA is defined in the feature spec. Even a soft indicator ("Bursars typically review requests within 1 school day") would reduce inbound calls.

> **Friction point — adjustment as override**: If the bursar adjusts the plan rather than approving it as requested (e.g., changes from 4 instalments to 6), the parent receives: *"Your plan for [Student] has been set up with a small adjustment."* From the parent's perspective, the school changed the terms they asked for without explanation. A parent who specifically requested 4 payments may have done so because that's what fit their budget — 6 smaller payments is not automatically better. "A small adjustment" is opaque; it does not tell the parent what changed or why. This is a trust moment: the product exists to give parents dignity in fee arrangements, and an unexplained change undercuts that. The adjustment SMS must frame the change as supportive and state the new terms explicitly. Recommended framing: *"To give you more time, your plan has been adjusted to [N] payments of KES [amount]. Next due: [date]. Pay to [paybill] as usual."* The reason and the revised schedule must both appear — not a generic "small adjustment."

### After
Parent has submitted a plan request without calling the school. They'll receive an SMS when it's approved, adjusted, or declined. Until then, they can check the app and see a PENDING_APPROVAL status against the student's fee balance.

## J3 — Parent Requests a Plan via SMS (Path B)

**Actor**: Parent
**Mental state entering**: Similar to J2, but this parent either doesn't have the app, prefers SMS, or was told about the PLAN keyword by the school. SMS journeys must complete in minimal messages — parents abandon multi-step SMS flows quickly.

### Trigger
Parent sends **PLAN** to the school's inbound SMS number. (Depends on OQ-06 — inbound SMS infrastructure must be confirmed.)

### Journey Steps

**1. Sending PLAN**
Parent sends the keyword. No navigation, no login. This is the dignity-first path: the lowest possible friction to make a plan request.

**2. Multi-student resolution (if applicable)**
If the phone number has multiple students linked: *"Reply with the student number: 1. [Student A] 2. [Student B]"*

This is a potential drop-off point. Parents with two children in the same school may not know their child's "student number" in Zeraki's system — the message should use student names, not internal IDs.

**3. Receiving template options**
*"[School name] payment plans for [Student name]: Reply 1 for [Template 1] | Reply 2 for [Template 2]..."*

160 characters. Template names must be short enough to fit alongside the framing copy. Long template names (e.g., "Monthly payment over 3 months") may truncate.

**4. Receiving and reviewing the schedule**
*"Plan: [N] payments of KES [amount]. Due dates: [date 1], [date 2]... Reply YES to confirm or NO to cancel."*

The parent is reading this on a low-end phone, possibly in a noisy environment. The dates and amounts must be in the simplest possible format. "Apr 4, Apr 18" is more readable than "2026-04-04, 2026-04-18".

**5. Replying YES / NO**
YES → plan request created, SMS confirmation sent.
NO → *"No plan has been set up. You can start over by sending PLAN again."*

The NO confirmation is correct: non-committal, leaves the door open, no penalty for changing your mind.

**6. Session expiry (10-minute window)**
If the parent doesn't respond within 10 minutes, the session expires silently. No SMS is sent. The parent can start over by sending PLAN again.

> **Friction point — silent session expiry**: If a parent is distracted (real-life scenario: they sent PLAN, then their child needed attention) and comes back 15 minutes later to reply, there's no indication the session has expired. They'll get a confused response or no response. Consider whether a session-expiry SMS ("Your session has expired. Send PLAN to start again.") would reduce confusion, or whether it creates more noise than value.

> **Friction point — adjustment as override**: Identical to J2. If the bursar adjusts the plan the parent requested via SMS, the parent receives *"set up with a small adjustment"* — with no indication of what changed or why. The SMS channel makes this worse: the parent cannot see the original request or the new terms side by side. The adjustment SMS must state the new schedule explicitly and frame the change as supportive: *"To give you more time, your plan has been adjusted to [N] payments of KES [amount]. Next due: [date]. Pay to [paybill] as usual."* This is especially important for SMS-initiated plans, where the parent went through a multi-step flow to select a specific template and confirm a specific schedule.

### After
Same outcome as J2: PENDING_APPROVAL request exists, confirmation SMS sent, bursar sees the queue item. The parent's entry cost was two or three SMS replies.

## J4 — Bursar Creates a Plan Directly (Path C — In-Person)

**Actor**: Bursar
**Mental state entering**: A parent has just left the bursar's office — or is still sitting across from them. An arrangement has been agreed verbally. The bursar needs to record it immediately before the next person comes in.

### Trigger
Bursar opens the student's fee record and selects **Create Plan** immediately following (or during) an in-person conversation with a parent.

### Journey Steps

**1. Opening the student record**
Bursar is already in Zeraki Finance — this is their normal working environment. The **Create Plan** action needs to be accessible from the student fee record without requiring extra navigation. The bursar's attention is split between the screen and the parent.

**2. Choosing template or custom plan**
Two options: use a configured template, or enter a custom plan.

- **Template**: Fastest path. Bursar sees the same templates parents see, with pre-calculated schedules. For the common case (parent agrees to the standard plan), this is a two-click confirmation.
- **Custom plan**: Bursar enters amount, number of instalments, and specific due dates. This path is for edge cases — a parent who needs a non-standard arrangement the templates don't cover. The custom form should not be the default focus; templates should be prominent.

> **Friction point — custom plan validation**: The bursar may enter due dates that extend beyond the term end date. The system must catch this inline (not on submit) — a bursar in an in-person conversation does not want to re-navigate a form.

**3. Reviewing and confirming**
Bursar reviews the calculated schedule. If the parent is still present, the bursar may turn the screen to show them. This is a real scenario — the UI should render legibly at arm's length.

**4. Plan created as ACTIVE**
No approval queue. The bursar is the approver by definition. Plan goes live immediately.

**5. Parent SMS sent**
Parent receives the plan active SMS. If the parent is still in the room, this serves as their receipt. The SMS should be sent immediately on save — not batched.

### After
Bursar returns to the next person in the queue. Parent leaves with an active plan and an SMS receipt. The arrangement that was verbal is now recorded.

## J5 — Bursar Processes the Approval Queue

**Actor**: Bursar
**Mental state entering**: The queue is part of the daily routine, but it's also a judgment call for each item. Bursars are cautious — a wrong approval or decline has a real human consequence. They want to act quickly but correctly.

### Trigger
Bursar opens the Instalment Plans section and sees pending plan requests. May be prompted by a notification (OQ-10 — notification channel unresolved).

### Journey Steps

**1. Arriving at the queue**
The queue shows: student name, parent name, requested template, calculated schedule, outstanding balance. The bursar is scanning for anything unusual — a very large balance, an unfamiliar parent name, a request from a student who just made a complaint.

The list density matters: bursars at large schools may see 20+ requests at once. The design must support rapid scanning, not card-by-card reading.

**2. Reviewing a single request**
For each request, the bursar makes one of three decisions:

**Approve** — the plan is as requested, nothing to change. One click. This should be the most visually accessible action. The majority of queue items in a functioning school will be approved.

**Adjust** — the bursar wants to modify the schedule. The plan editor opens pre-filled with the parent's requested schedule. The bursar can change amounts, dates, or instalment count. This is an edit-then-approve flow, not a reject-and-resubmit loop — the parent gets an active plan, just a slightly different one.

> **Friction point — adjustment without context**: The bursar may want to note why they adjusted, but the spec does not include an adjustment reason field. Consider whether this is needed for audit purposes, or whether the SMS ("set up with a small adjustment") is sufficient.

**Decline** — the plan request cannot be approved. Bursar optionally enters a reason. The reason flows directly into the SMS: *"could not be approved: [reason]."*

> **Friction point — reason quality**: If the bursar enters a terse or internal note ("too many missed before"), the parent receives it verbatim. Either the reason field should be parent-facing (prompt: "This message will be sent to the parent") or it should be an internal note with a separate parent-facing message.

> **Friction point — decline as dead end (parent experience)**: This is the highest-emotion moment in the product for the parent. They chose the dignified digital path — requested a plan through the app or SMS rather than calling or visiting — and received a rejection that tells them to "contact the school." That redirects them back to the exact face-to-face gate negotiation the product exists to prevent. The decline SMS as specified (*"could not be approved. Please contact the school."*) offers no concrete next step. The parent does not know who to call, what to ask for, or whether there is any alternative. **The decline SMS must include a concrete action**: either a direct contact for the bursar ("Call [phone number]"), a location ("Visit the school office"), or — if the school has configured it — an alternative arrangement. Without this, the decline lands as a rejection with no path forward, which damages both parent trust and the school's relationship with that family. Flag for the UI Spec Designer: the decline flow in the bursar's queue should prompt the bursar to enter a contact detail or next-step instruction that is included in the SMS alongside any reason.

**3. Bulk approve (Approve All)**
For bursars with a long queue and no flags, **Approve All** is the right escape valve. It should be prominent but not the first thing the eye goes to — reviewing at least once is the intended default.

### After
Queue items are resolved. Each approved parent has an active plan and has received an SMS. Declined parents have been notified. Bursar moves on to other work.

## J6 — Parent Receives Instalment Reminder (System-Initiated)

**Actor**: Parent
**Mental state entering**: Not in any product interface — this arrives as an SMS. Parent may be at work, at home, or elsewhere. The message competes with all other SMS notifications.

### Trigger
Scheduled system job identifies an instalment due within the reminder window (OQ-01 — lead time unresolved) and sends reminder SMS.

### Journey Steps

**1. Receiving the SMS**
*"[School]: Reminder — KES [amount] due for [Student] on [date]. Pay to [paybill] as usual."*

The message is functional and complete. The parent knows what to pay, when, and where. No action is required inside any app or system.

**2. Parent's response paths**
- **Pays before due date** → instalment becomes PAID_ON_TIME (Flow 7 / J7)
- **Pays after due date** → instalment becomes PAID_LATE
- **Does not pay within grace period** → instalment becomes MISSED (J9)
- **Already paid in full** → reminder is redundant but harmless. Parent may call the school to report it.
- **Partially paid** → this is the critical case. If a parent has already paid KES 2,000 of a KES 3,750 instalment, a reminder that says *"KES 3,750 due for [Student] on [date]"* is factually wrong from the parent's perspective. They made a payment. The system acknowledged it. Getting a reminder for the original full amount feels like the school didn't notice — or worse, like their partial payment didn't count. **The reminder SMS must reference the remaining instalment balance, not the original amount.** Correct: *"KES 1,750 still due for [Student] on [date]."* Flag for the UI Spec Designer: this is a data requirement — the reminder job must read `instalment.amount - instalment.paid_amount` as the reminder figure, not `instalment.amount`.

### After
Parent either acts on the reminder or doesn't. The journey ends in their hands.

> **Design note — reminder window (OQ-01)**: The reminder lead time directly shapes this journey. A 1-day reminder is urgent; a 7-day reminder is planning-oriented. The right default depends on how many parents are living paycheck-to-paycheck (short lead time is more actionable) vs. how much time is needed to arrange an M-Pesa payment (usually immediate). Recommend a 3-day default with school-level configurability.

## J7 — Payment Auto-Matched to Active Plan

**Actor**: Parent (passive); Bursar (notified)
**Mental state entering**: Parent has just made an M-Pesa payment. They expect confirmation — M-Pesa's own SMS always comes first, then Zeraki's. The parent is looking for the second SMS to confirm the plan is tracking their payment.

### Trigger
M-Pesa paybill webhook arrives for a student with an ACTIVE plan.

### Journey Steps

**1. System allocates payment**
FIFO. Oldest PENDING instalment allocated first. No parent action required.

**2. Parent receives payment received SMS**
*"[School]: KES [amount] received for [Student]. Remaining plan balance: KES [amount]. Next due: KES [amount] on [date]."*

This is the most frequent SMS event in the product's lifecycle — a parent on a 4-instalment plan will receive 4 of these. The message should be scannable: received, remaining, next. The current spec achieves this.

**3. Plan completion path**
If this payment clears the last instalment: system sends plan completion SMS. *"[Student]'s payment plan is complete. All instalments paid. Thank you."*

This SMS should feel like a small celebration — not a receipt. The word "complete" is correct. Consider "Thank you" as a closing note (already in the spec). No further action needed.

**4. Bursar dashboard updates**
The bursar sees the updated plan status without opening anything. The dashboard is their passive oversight layer.

### After
Payment is recorded, adherence metrics updated, parent notified. If on-time: on-time rate ticks up. If late: completion rate only. If plan is now COMPLETED: both parties get closure.

## J8 — Bursar Records a Manual Payment (Cash / Bank Deposit)

**Actor**: Bursar
**Mental state entering**: A parent paid in cash at the office, or brought a bank deposit slip. The bursar needs to record this now — before the next parent arrives, before the receipt book is misplaced, and definitely before the system auto-marks the instalment MISSED.

### Trigger
Bursar opens the student's fee record and selects **Record Payment**.

### Journey Steps

**1. Entering payment details**
Form fields: amount, date (defaults to today), payment method (Cash / Bank deposit), optional reference number.

The date field is the critical one. The spec allows backdating — the bursar enters the actual payment date, not the entry date. This is correct for on-time/late determination (BR-17). But it creates audit risk if bursars routinely backdate to improve adherence scores.

> **Friction point — backdating visibility**: Consider whether the UI should surface a visual indicator when the entered date differs from today's date ("Recording payment for 3 days ago — this will be treated as received on [date]"). Not a blocker, just a confirmation moment for the bursar.

**2. Duplicate detection (before applying)**
The system checks for a possible duplicate before applying the payment. If a match is found, the bursar sees:

*"A [M-Pesa / manual] payment of KES [amount] was already recorded for [Student name] on [date]. Is this the same payment or a separate one?"*

Two options: **Dismiss this entry** or **Apply both**.

This interruption is necessary. The bursar may be flustered if they didn't know the parent had also paid via M-Pesa. The resolution prompt needs to be calm, specific, and not make the bursar feel like they made a mistake — they may not have.

However, the two-button prompt puts a data-integrity decision in front of someone who may not understand the downstream consequence of getting it wrong. A bursar choosing between "Dismiss" and "Apply both" without context may select incorrectly — applying a duplicate and inflating adherence metrics, or dismissing a legitimate second payment. Each option must carry a one-line explanation of what it does:

- **Dismiss this entry** — *This was the same payment recorded twice. Only the original will be applied.*
- **Apply both** — *The parent made two separate payments. Both will be applied to the plan.*

These explanations make the choice about the parent's real-world action, not about the system state. The bursar can answer "did the parent pay once or twice?" — they cannot reliably answer "should I dismiss or apply both?"

> **Friction point — resolution responsibility**: "Apply both" makes the bursar explicitly responsible for overriding the duplicate flag. This is correct from a data-integrity standpoint. The UI must make clear this is an informed decision: a soft audit trail note ("Applied both — confirmed by [bursar name] on [date]") is appropriate for the plan's payment history.

**3. Payment applied**
Same FIFO allocation logic as Flow 7. Bursar sees updated instalment status immediately. Parent receives the same payment received SMS as an M-Pesa payment would trigger.

### After
Cash-paying parent is correctly recorded. No adherence gap created by payment channel. Bursar has completed the task without leaving the student record.

## J9 — Instalment Marked MISSED — Parent and Bursar Notified

**Actor**: Parent (receives overdue SMS); Bursar (receives alert SMS + sees dashboard flag)
**Mental state entering**: Parent may have forgotten, may be unable to pay, or may have already paid and the record is incorrect (Cash payment not entered by bursar, M-Pesa webhook failure). Bursar is the triage point.

### Trigger
Scheduled system job: instalment past due date by more than [grace period] days (OQ-02) with no full payment received.

### Journey Steps

**1. Parent receives overdue SMS**
*"[School]: KES [amount] for [Student] was due on [date] and has not been received. Please pay to [paybill] or contact the school."*

Tone must be firm but non-threatening. The spec copy achieves this. The "contact the school" option is important — it gives the parent an out if they've already paid through a channel the system didn't capture.

**2. Bursar receives alert SMS**
*"[Student name]: KES [amount] instalment due [date] has not been paid. Review in Zeraki Finance."*

This is a per-student alert. Bursars at large schools may receive many of these if several plans are in arrears. The alert triggers a review, not an action — the bursar decides whether to call the parent, record a missed cash payment, or note a genuine default.

> **Friction point — alert fatigue**: If 30 students have missed instalments simultaneously (e.g., post-school holiday), the bursar receives 30 SMS alerts. Consider whether a digest ("5 instalment payments are overdue — review in Zeraki Finance") is preferable to individual alerts for high-volume scenarios, or whether individual alerts are required for the bursar to identify specific students.

**3. Bursar reviews the dashboard**
Overdue instalments are surfaced in the dashboard (Flow 11 / J11). Bursar can drill into individual plans. No automated escalation — the bursar takes the next step.

### After
Both parties are notified. Adherence metrics have been updated (completion rate unaffected; on-time rate already cannot improve for this instalment). The plan remains ACTIVE — the student is not sent home. Only the bursar can cancel or escalate.

## J10 — Parent Checks Fee Balance via SMS (BAL Keyword)

**Actor**: Parent
**Mental state entering**: Quick information need. Parent wants to know what's owed and when the next payment is due without calling the school. This is a low-stakes, high-frequency interaction.

### Trigger
Parent sends **BAL** to the school's inbound SMS number.

### Journey Steps

**1. Sending BAL**
One word. Zero navigation. The lowest-friction query in the product.

**2. Receiving balance SMS**
*"[Student]: Fees due KES [amount]. Plan active: Yes. Next instalment KES [amount] due [date]."*

If multiple students: the system returns a summary for each. This could become long — 3 students with active plans is 3 response lines, potentially a multi-part SMS.

**3. Parent acts on the information**
Either pays (triggers J7), calls the school with a question, or does nothing. The BAL response is information, not a call to action.

> **Design note — BAL with no active plan**: If no plan exists, the response is: *"Plan active: No"* — which is correct but may leave the parent wondering how to set one up. Consider whether the BAL response should include a hint ("Send PLAN to request a payment plan") when no plan is active and a fee balance exists.

### After
Parent has the information they needed. No bursar involvement. This is the self-service path — every BAL response that gives a clear answer is a phone call to the school that didn't happen.

## J11 — Bursar Reviews the Instalment Plan Dashboard

**Actor**: Bursar
**Mental state entering**: This is the monitoring journey — the bursar is not here to take a specific action but to understand the current state of fee collection and identify anything that needs attention. Morning check, end-of-day review, or triggered by a concern.

### Trigger
Bursar navigates to the Instalment Plans section.

### Journey Steps

**1. Term-level summary**
Dashboard shows:
- Total active plans, KES expected, KES received
- Plans fully on track / PAID_LATE / MISSED
- Pending approval queue count (one-click access)
- Upcoming instalments due in the next 7 days
- Overdue instalments

The bursar's primary scan is for anomalies: pending queue items that are getting stale, a spike in MISSED instalments, or a large upcoming obligation the school isn't ready for. The summary should surface these, not bury them in equal-weight metrics.

**2. Drilling into individual plans**
Bursar selects a student's plan to see the full instalment schedule, payment history, and both adherence metrics (on-time rate, completion rate).

> **Design note — adherence metric visibility**: On-time rate and completion rate are the feature's primary output for Layer 2 credit eligibility (see Cross-Feature Dependencies). Bursars should be able to see and understand these metrics — but they also shouldn't feel like the bursar is being graded. The framing should be: "Here's how this plan is progressing" not "Here's this parent's credit score input."

**3. Queue access from dashboard**
Pending approval queue is one click from the dashboard header. This prevents bursars from needing to navigate away to process new requests — queue management and monitoring are in the same section.

### After
Bursar has a current picture of fee collection via plans. Any actions triggered — processing queue items, recording a manual payment, cancelling a plan — happen as sub-journeys from here.

## J12 — Bursar Cancels a Plan

**Actor**: Bursar
**Mental state entering**: Something has changed — a student has left the school, a parent has paid the full balance outside the plan, or the original arrangement no longer makes sense. This is a deliberate, uncommon action.

### Trigger
Bursar opens an active student plan and selects **Cancel Plan**.

### Journey Steps

**1. Selecting Cancel Plan**
The **Cancel Plan** action should not be in the primary action hierarchy — it's a destructive action on an active record. It should require deliberate navigation (e.g., a "More actions" menu or a secondary position in the plan detail view).

**2. Confirmation prompt**
*"Cancelling this plan will remove all remaining scheduled instalments. Payments already received will remain recorded. Are you sure?"*

This prompt does three things right:
1. States the consequence precisely (remaining instalments removed)
2. Reassures the bursar that history is preserved (payments already received)
3. Asks for explicit confirmation

The confirm button should restate the action: **Cancel this plan** — not "Yes" or "Confirm".

**3. Plan cancelled**
System transitions to CANCELLED. All PENDING instalments marked CANCELLED. PAID_ON_TIME, PAID_LATE, and MISSED records are preserved and immutable.

**4. Parent SMS sent**
*"[School]: Your payment plan for [Student] has been cancelled. Please contact the school to discuss your fee balance."*

This SMS is correct: it doesn't explain why (that's a phone conversation), and it directs the parent to the right next step.

### After
The plan is cancelled. The parent has been notified. The fee balance reverts to an unstructured outstanding amount — no plan, no due dates, no automated reminders. The school and parent need to make a new arrangement if required.

## Journey Patterns Established (update conventions/journey-patterns.md)

The following patterns emerged from these journeys and should be added to `conventions/journey-patterns.md`:

### Trigger Patterns
| Trigger Type | Description | Established in |
|-------------|-------------|----------------|
| Self-initiated (in-person handoff) | Actor takes a product action immediately following a real-world conversation — partial attention, time pressure | J4 — Bursar creates plan after in-person conversation |
| System-prompted (approval queue) | Product surfaces a work queue the actor needs to process | J5 — Bursar processes approval queue |
| Inbound SMS keyword | Actor sends a keyword to initiate a self-service flow — zero navigation, zero auth | J3 — PLAN keyword, J10 — BAL keyword |

### Friction Resolution Patterns
| Friction Type | Resolution Pattern | Established in |
|--------------|-------------------|----------------|
| Commitment anxiety (non-destructive) | Show the commitment prominently before the confirm action; include it in post-activation confirmation | J1 — school opt-in commitment statement |
| Abstract configuration (structure vs. value) | Explain calculation method inline: "amounts are calculated per student at request time" | J1 — template amount confusion |
| Waiting after submission (no SLA) | Surface expected processing time as a soft indicator | J2 — parent waits for plan approval |
| Silent session expiry (SMS) | Consider session-expiry SMS so actors can restart; balance against notification noise | J3 — 10-minute SMS session expiry |
| Alert fatigue (bulk system events) | Evaluate digest vs. per-item alerts for high-volume scheduled jobs | J9 — multiple missed instalment alerts |
| Destructive action on active record | Use secondary action position (not primary CTA); restate action in confirm button | J12 — Cancel Plan |

### "After" Patterns
| After Type | Description | Established in |
|-----------|-------------|----------------|
| SMS as receipt | A system event SMS serves as the parent's record of a real-world transaction (plan approval, payment received) | J4, J7, J8 |
| Self-service deflection | A successful BAL/PLAN SMS response is a school call that didn't happen | J3, J10 |
| Active record + passive oversight | Bursar completes a transaction; dashboard reflects the change without requiring the bursar to navigate there | J7, J8, J11 |

## Open Questions Surfaced by Journey Design

The following questions from the feature spec directly affect journey quality and should be prioritised:

| OQ | Impact on journey |
|----|------------------|
| OQ-01 — Reminder lead time | Shapes whether the reminder is a planning prompt or an urgent nudge (J6) |
| OQ-02 — Grace period | Determines how quickly the MISSED state is entered and how much time parents have to catch up after a missed due date (J9) |
| OQ-05 — Parent app existence | If the app is a new build, Path A (J2) is a significant scope addition; SMS-only (J3) may be the MVP path |
| OQ-06 — Inbound SMS infrastructure | Path B (J3) and BAL keyword (J10) are blocked without inbound SMS capability |
| OQ-10 — Bursar queue notification | J5 depends on the bursar being alerted to queue items — without a notification, queue items may age unresolved |
