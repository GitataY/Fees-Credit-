## Screen 1: Settings — Fee Top-Up Credit

**Purpose**: Allow the Headteacher / School Admin to understand what Fee Top-Up Credit does, accept the paybill commitment, and enable the feature — or be redirected to Layer 1 settings if Instalment Plans are not yet active.
**Actor**: Headteacher / School Admin
**Triggered by**: Navigate to Settings → Fee Collection → Fee Top-Up Credit
**Journey reference**: J1 — School Enables Fee Top-Up Credit

---

### Layout Intent

**Structure**: Full-width settings page within the Zeraki Finance shell. Single centered content column. Mirrors the layout structure of the Layer 1 settings screen (1.0.1 Screen 1) — both live in the same settings section. Persistent Finance shell header and sidebar.
**Visual hierarchy**: Risk clarification callout (answers the admin's first question before they read anything else) → feature description → commitment card → Enable button. Post-enable: confirmation message only — no configuration step required.
**Key regions**:
- **Page header**: "Fee Top-Up Credit" + breadcrumb (Settings → Fee Collection → Fee Top-Up Credit)
- **Layer 1 dependency gate** (conditional): Shown only if Instalment Plans are not enabled. Blocks all further content.
- **Risk clarification callout**: A prominent one-liner that answers the admin's primary concern before they read the commitment. Not buried below the fold.
- **Feature description region**: Three-point explanation of how the product works
- **Commitment card**: The paybill routing commitment + concrete example
- **Enable button**: Only shown when Layer 1 is active

---

### Component Inventory

#### PageHeader — Navigation
**Purpose**: Orient the admin in the settings hierarchy
**Copy**:
- Breadcrumb: "Settings / Fee Collection / Fee Top-Up Credit"
- Page title: "Fee Top-Up Credit"

---

#### Layer1DependencyGate — AlertBanner (blocking)
**Purpose**: Redirect the admin to enable Instalment Plans before Fee Top-Up Credit can be activated
**Copy**:
- Message: "Fee Top-Up Credit requires Instalment Plans to be active because it uses plan adherence data to assess eligibility for some parents. Enable Instalment Plans first, then return here."
- CTA: "Go to Instalment Plans settings" → links to Screen 1 of 1.0.1

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| visible | Layer 1 not enabled | Info/warning banner at top of content area; all other screen content hidden |
| hidden | Layer 1 is enabled | Not rendered; full screen content visible |

---

#### RiskClarificationCallout — CommitmentCard (info variant)
**Purpose**: Answer the admin's first question — "does the school owe money if a parent defaults?" — before they read anything else
**Copy**: "The school takes no financial risk. The lending partner provides the credit and holds the default risk."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible when Layer 1 is active | Elevated card, info tone (not warning); positioned above the feature description. Short, single sentence — not a paragraph. |

> **Design note (J1)**: This callout must appear *before* the feature description and commitment card, not after. The admin's trust barrier is financial risk. Clear it first; everything else reads differently once it is cleared.

---

#### FeatureDescriptionBlock — Content Block
**Purpose**: Explain how the product works — the three things an admin needs to understand before committing
**Copy**:
- Headline: "Fee support, funded by a lending partner"
- Point 1: "Eligible parents are identified by Zeraki's payment history analysis — the school does not select or recommend anyone."
- Point 2: "The lending partner credits the fee amount directly to your school's paybill."
- Point 3: "Repayment happens automatically: when parents pay fees, a portion is routed to the lending partner. No new payment channel. No school action required."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Layer 1 active, feature not yet enabled | Visible; neutral content block |

---

#### CommitmentCard — Container / Callout
**Purpose**: Surface the paybill routing commitment with a concrete example so the admin understands what "automatically split" means before they enable
**Copy**:
- Label: "School commitment"
- Body: "By enabling Fee Top-Up Credit, this school agrees that its M-Pesa paybill will be used as the repayment routing channel — incoming fee payments from parents with active credit arrangements will be automatically split between the school's fee balance and the lending partner's repayment share."
- Example label: "What this looks like in practice:"
- Example body: "A lending partner credits KES 4,500 to your paybill for a student's fee top-up. When that parent later pays KES 2,000 in fees, a portion goes to repay the credit advance and the remainder is credited to the student's school fee balance. The school received the KES 4,500 upfront — repayment routing returns it to the lender over time."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Visible before enabling | Elevated card, left border accent (info/warning tone). Commitment text full-size — not small print. Example rendered below in slightly smaller but fully readable type. |
| acknowledged | After enabling | Card remains visible as a record; border muted |

> **Design note (J1)**: The concrete example is load-bearing. "Automatically split" is abstract; the example makes it legible. The example must appear inside the CommitmentCard, not in a tooltip or expandable — the admin needs to see it before clicking Enable.

> **Design note — net-neutral framing (J1)**: The admin may read "payment is split" as income the school loses. The example copy above frames the chain correctly: (1) lender disburses → school gets cash upfront, (2) future fee payments return lender's share. Net result to school is positive. This framing is required in the example copy — do not simplify it away.

---

#### EnableFeeTopUpCreditButton — PrimaryButton
**Purpose**: Opt the school into Fee Top-Up Credit after reading the commitment
**Copy**: "Enable Fee Top-Up Credit"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| hidden | Layer 1 not enabled | Not rendered (Layer1DependencyGate is shown instead) |
| default | Layer 1 active; feature not yet enabled | Enabled; positioned below CommitmentCard |
| loading | Clicked; system activating | Spinner, "Enabling…", disabled |
| hidden (post-enable) | Feature now enabled | Not rendered; replaced by confirmation message |

---

#### EnabledConfirmationBlock — AlertBanner (success) + Content Block
**Purpose**: Confirm activation and orient the admin to what happens next — without requiring any further action
**Copy**:
- Banner: "Fee Top-Up Credit is now enabled."
- Body line 1: "Eligible parents will be identified at the start of each term and offered fee support."
- Body line 2: "Offers go out under your school's sender ID, but eligibility is determined by Zeraki's payment analysis — not by the school. The school does not select or recommend any individual parent."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| hidden | Feature not yet enabled | Not rendered |
| visible | Feature enabled | Success banner + body text; replaces EnableFeeTopUpCreditButton |

---

|-------|---------|----------------------|
| layer-1-blocked | Admin arrives; Layer 1 not active | Layer1DependencyGate shown; all other content hidden |
| disabled | Layer 1 active; feature off | Full screen content visible; RiskClarificationCallout + FeatureDescriptionBlock + CommitmentCard + EnableButton |

## Screen 3: Offer Screen (Parent App)

**Purpose**: Explain the fee top-up offer — amount, how it works, and repayment terms — so the parent can make an informed decision before entering the consent flow.
**Actor**: Parent
**Triggered by**: Tap offer banner on Screen 2 or push notification
**Journey reference**: J2 — Steps 1 & 2

---

### Layout Intent

**Structure**: Mobile-first, single-column scroll. Top: offer headline with student name and credit amount (visually dominant). Middle: how-it-works explanation + repayment schedule. Bottom: primary CTA to begin consent + "No thanks" decline.
**Visual hierarchy**: Credit amount + student name (largest, most prominent) → "Repaid through your regular fee payments" (repayment framing, must appear before consent) → repayment schedule → Continue CTA
**Key regions**:
- **Screen header**: Back arrow + school name (not "Zeraki" — school trust signal)
- **Offer headline block**: "KES [amount] fee support available for [Student name]" — amount and name are the dominant visual elements
- **How-it-works explanation**: Credited to school; repaid through regular fee payments; no new payment channel. Three short statements.
- **Repayment schedule block**: Estimated per-week repayment + total repayment amount in KES (BR-17: KES total, not percentage)
- **Waterfall expectation callout**: Sets expectation that future fee payments will be split
- **Action bar**: "Continue" (primary) + "No thanks" (secondary/text link)

---

### Component Inventory

#### OfferHeadlineBlock — Content Block
**Purpose**: Lead with the credit amount and student name — the two most decision-relevant pieces of information
**Copy**:
- Headline: "KES [amount] fee support for [Student name]"
- Subhead: "Credited directly to [School name]"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Data fetching | Skeleton — headline placeholder |
| populated | Offer loaded | Amount in large type, student name prominent below |

---

#### HowItWorksBlock — Content Block
**Purpose**: Explain the mechanics clearly before the parent reaches consent — progressive disclosure, not concealment
**Copy**:
- "The fee support amount is credited directly to [School name] on your behalf."
- "You repay through your regular fee payments — part of each payment you make will go toward repaying the support."
- "Nothing changes about how you pay. Same paybill, same process."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible on this screen | Body text, readable size — not small print |

> **Critical design rule (J2)**: The phrase "You repay through your regular fee payments" must appear on this screen — before Consent Step 1. A parent who thinks this is a grant will feel misled when they reach CRB disclosure. This is progressive disclosure done right: the parent sees "repayment" before they encounter the legal consent layer.

---

#### RepaymentScheduleBlock — Data Display
**Purpose**: Show the estimated repayment schedule in parent-legible terms — what to expect each week
**Copy**:
- Section label: "Estimated repayment"
- Per-period label: "Approximately KES [amount] per [week / month]"
- Total label: "Total to repay: KES [credit_amount + flat_fee]"
- Note: "Exact amounts depend on each payment you make."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Calculating | Skeleton rows |
| populated | Schedule calculated | Per-period and total amounts visible |

> **Design rule (J2, BR-17)**: Total repayment must be shown as a KES figure — not as "5% flat" or any percentage. "KES 225 on top of KES 4,500" is legible; "5% flat" is not, for parents anchoring on absolute KES amounts.

---

#### WaterfallExpectationCallout — CommitmentCard (soft variant)
**Purpose**: Set the waterfall repayment expectation at acceptance — the primary mitigation for J6 confusion
**Copy**: "When you make a fee payment, part will go toward repaying your fee top-up and the rest will go to your school balance. You'll receive an SMS after each payment showing the split."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible on this screen | Subtle callout card — informational, not alarming. Muted border. |

> **Journey note (J2)**: This callout is the most important UX intervention in the entire feature. Every J6 confusion trace back to whether this expectation was set here. It must not be omitted, hidden behind a toggle, or placed below the fold.

---

#### ContinueToConsentButton — PrimaryButton
**Purpose**: Proceed to Consent Step 1
**Copy**: "Continue"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Offer loaded | Enabled |
| disabled | Offer still loading | Disabled |

---

#### NoThanksLink — SecondaryButton / Text Link
**Purpose**: Allow the parent to decline the offer without friction — no guilt UI
**Copy**: "No thanks"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible | Low-emphasis; does not compete with Continue. Does not say "Decline" or "Reject" — "No thanks" is the correct label (parent's autonomy respected, not their decision framed as an error). |

> **Hard design rule (J4)**: No "Are you sure? You might miss out" confirmation on this decline path. No guilt patterns. The parent taps "No thanks" → arrangement transitions to DECLINED → offer disappears from the fee screen. Done.

---

### Screen States

| State | Trigger | What changes on screen |
|-------|---------|----------------------|
| loading | Screen opens | OfferHeadlineBlock + RepaymentScheduleBlock show skeletons |
| populated | Offer and schedule loaded | Full content visible; ContinueToConsentButton enabled |
| declined | "No thanks" tapped | No confirmation dialog. Navigate back to Screen 2; OfferBanner removed. |
| error | Offer data fails to load | AlertBanner: "Could not load your offer details. Please try again." |

---

### Transitions
**On Continue**: Navigate to Screen 4 — Consent Step 1
**On No thanks**: Decline arrangement; navigate to Screen 2; OfferBanner no longer shown

## Screen 4: Consent Step 1 (Parent App)

**Purpose**: Disclose data sharing with the credit partner and CRB reporting. The parent must explicitly acknowledge both before proceeding.
**Actor**: Parent
**Triggered by**: Tap "Continue" on Screen 3
**Journey reference**: J2 — Step 3 (Consent Step 1)

---

### Layout Intent

**Structure**: Mobile-first, single-column. Top: screen title + progress indicator (Step 1 of 2). Middle: disclosure text — prominent, not small print. Below: CRB framing explanation. Bottom: acknowledge CTA + back.
**Visual hierarchy**: "Before you continue" heading → disclosure statements (clear, full-size) → CRB framing (contextual, helpful) → "I understand — continue" CTA
**Key regions**:
- **Screen header**: Back arrow + "Fee support — step 1 of 2"
- **Disclosure block**: Two explicit statements about data sharing and CRB reporting
- **CRB framing block**: Reframes CRB reporting as a consequence of behaviour (positive if repaid; only negative if missed) — addresses the anxiety the journey predicts
- **Expandable detail** (optional): "What does this mean?" — if the framing block is insufficient. Flag as open UI question.
- **Acknowledge CTA**: "I understand — continue"
- **Back link**: Returns to Screen 3 without recording any consent

---

### Component Inventory

#### ConsentStep1Header — SectionHeader
**Copy**:
- Screen title: "Before you continue"
- Subtitle: "Step 1 of 2"

---

#### DataSharingDisclosureBlock — Content Block
**Purpose**: State clearly what data is shared and with whom — required for informed consent
**Copy**:
- Statement 1: "To receive this offer, [School name] will share your fee payment history with our credit partner."
- Statement 2: "Repayment activity will be reported to credit reference bureaus."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible on this screen | Full-size body text — not small print. Each statement on its own line with a checkbox or visual indicator that it has been displayed (not an interactive checkbox — display only, to confirm the parent saw it before the CTA becomes active). |

> **Compliance note (J2)**: Consent text must be reviewed against CBK Digital Credit Provider Regulations 2022 and Kenya Data Protection Act 2019 before launch. Consent version must be stored on the arrangement record. The exact legal text may differ from the copy above — the design must accommodate longer legal copy without truncating or hiding it. Flag as OQ-UI (legal review dependency).

---

#### CRBFramingBlock — Content Block
**Purpose**: Reframe CRB reporting as a consequence of behaviour — remove the threat framing; restore accurate framing
**Copy**:
- "Repaying on time builds your credit history with credit reference bureaus — a positive record."
- "CRB reporting only has negative consequences if repayments are missed."
- "The data shared is your fee payment history at [School name] — not your general financial data."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible | Slightly muted from disclosure block — secondary information, but not hidden. Uses the same font size — muting is tonal, not typographic. |

> **Design note (J2)**: High abandonment at this step is the expected failure mode if this screen is poorly executed. The disclosure cannot be softened — both statements are real commitments. But the CRBFramingBlock changes the emotional register from "threat" to "consequence of your behaviour." Both must be on screen together; neither is sufficient alone.

---

#### IUnderstandContinueButton — PrimaryButton
**Purpose**: Record acknowledgement of Step 1 disclosure and proceed to Consent Step 2
**Copy**: "I understand — continue"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Disclosure visible | Enabled |
| loading | Processing | Spinner, "Continuing…", disabled |

> **Design rule**: "I understand — continue" is the exact copy from the feature spec (Flow 4, Step 2). Do not shorten to "Continue" — the "I understand" prefix signals that this is an acknowledgement step, not just navigation.

---

### Screen States

| State | Trigger | What changes on screen |
|-------|---------|----------------------|
| default | Screen opens | Disclosure block + CRB framing visible; CTA enabled |
| loading | CTA tapped | Button loading state |
| error | System error on proceed | AlertBanner: "Something went wrong. Please try again." |

---

### Transitions
**On I understand — continue**: Navigate to Screen 5 — Consent Step 2
**On Back**: Return to Screen 3 — Offer Screen; no consent recorded; arrangement remains OFFERED

## Screen 5: Consent Step 2 (Parent App)

**Actor**: Parent
**Triggered by**: Tap "I understand — continue" on Screen 4
**Journey reference**: J2 — Step 4 (Consent Step 2)

---

**Visual hierarchy**: Confirmation statement (specific amounts are the dominant text) → Confirm CTA
**Key regions**:
- **Screen header**: Back arrow + "Fee support — step 2 of 2"
- **Confirmation statement**: The exact amounts — KES credit, KES approx. per-period repayment, student name. No supporting content needed on this screen — the parent has already read everything.
- **Action bar**: Confirm (primary) + Back (text link)

## Screen 6: Post-Acceptance Confirmation (Parent App)

**Purpose**: Confirm the parent's accepted arrangement and bridge the gap between ACCEPTED and DISBURSED — the parent knows what to expect next, including the waterfall split on their next payment.
**Actor**: Parent
**Triggered by**: Tap Confirm on Screen 5; arrangement is now ACCEPTED
**Journey reference**: J2 — Step 5 (post-acceptance), J10 (disbursement failure)

---

### Layout Intent

**Structure**: Full-screen confirmation state, single focus. Centered vertically and horizontally. Icon + headline + body copy + action.
**Visual hierarchy**: Status icon → Headline → Body (two callouts: processing status + waterfall reminder) → Back to fees action
**Key regions**:
- **Status icon**: Processing indicator — not a success checkmark yet (disbursement is in progress, not confirmed)
- **Headline**: Processing state ("Fee top-up request confirmed")
- **Processing status**: What's happening now and what SMS to expect
- **Waterfall reminder callout**: Final opportunity to set split-payment expectation before J6 surprise occurs

---

### Component Inventory

#### ProcessingStatusBlock — Content Block
**Purpose**: Bridge the ACCEPTED → DISBURSED gap so the parent doesn't check the school balance and assume nothing happened
**Copy**:
- Headline: "Fee top-up request confirmed"
- Body: "We're processing the fee credit to [School name]. You'll receive an SMS once it's been credited to the school."
- In-progress note: "Your fee balance may not update immediately — check back after receiving the SMS confirmation."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| processing | Disbursement pending | Spinner or animated processing icon (not a success checkmark — disbursement not yet confirmed). Neutral tone. |
| disbursed | Disbursement webhook received | Icon changes to checkmark; body text changes to: "KES [amount] has been credited to [School name]. Your school fee balance has been updated." |
| failed | DISBURSEMENT_FAILED received | Icon changes to alert; headline changes to "Fee top-up could not be processed" (see DisbursementFailedBlock below) |

---

#### WaterfallReminderCallout — CommitmentCard (soft variant)
**Purpose**: Final pre-payment expectation-setting — the parent must leave this screen understanding the waterfall before their next payment
**Copy**: "When you make your next fee payment to [paybill], part will go toward repaying your fee top-up and the rest will go to your school balance. You'll get an SMS after each payment showing the split."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| visible | Processing or disbursed state | Always shown in these states — muted callout card, informational tone |
| hidden | Disbursement failed | Not shown — irrelevant if no credit was disbursed |

> **Journey note (J2)**: This callout repeats the expectation from Screen 3. Repetition is intentional — the parent was in a consent flow between Screen 3 and now; they may not have retained the earlier framing. This is the last chance to set the expectation before the first payment is split.

---

#### DisbursementFailedBlock — Content Block (error state)
**Purpose**: Inform the parent of the failure with a correct escalation route — not a dead end
**Copy**:
- Headline: "Fee top-up could not be processed"
- Body: "We were unable to process your fee top-up at this time. Please contact Zeraki Finance support or ask your school's bursar to flag this to Zeraki."
- Note: "Your acceptance has been recorded. We'll investigate and follow up."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| hidden | Processing or disbursed | Not rendered |
| visible | DISBURSEMENT_FAILED | Error tone; replaces WaterfallReminderCallout |

> **Journey note (J10)**: The failure copy routes escalation to Zeraki — not "contact the school." The bursar cannot resolve a disbursement failure; Zeraki support can. The school is mentioned as a relay ("ask your bursar to flag this"), but resolution ownership is Zeraki's.

---

#### BackToFeesButton — SecondaryButton
**Purpose**: Return the parent to the fee screen
**Copy**: "Back to fees"

---

### Screen States

| State | Trigger | What changes on screen |
|-------|---------|----------------------|
| processing | Arrangement ACCEPTED; disbursement pending | ProcessingStatusBlock in processing state; WaterfallReminderCallout visible |
| disbursed | DISBURSEMENT_CONFIRMED webhook received | ProcessingStatusBlock in disbursed state; icon → checkmark; balance update note shown |
| failed | DISBURSEMENT_FAILED | ProcessingStatusBlock in failed state; DisbursementFailedBlock shown; WaterfallReminderCallout hidden |

---

### Transitions
**On Back to fees**: Navigate to Screen 2 — Fee Screen; fee balance may now reflect disbursed amount (if DISBURSED) or show in-progress state

## Screen 7: Credit Dashboard

**Purpose**: Give the bursar a read-only view of the school's Fee Top-Up Credit activity — total disbursed, active arrangements, and the ability to answer parent enquiries.
**Actor**: Bursar
**Triggered by**: Navigate to Fee Top-Up Credit section in Zeraki Finance sidebar
**Journey reference**: J9 — Bursar Views the Credit Dashboard

---

### Layout Intent

**Structure**: Full-width within Finance shell. Page header with term selector. Top section: hero metric (disbursed amount, visually dominant) + secondary metric strip below it. Middle: at-risk callout (if any AT_RISK arrangements exist). Main content: ArrangementListTable with per-row drill-down.
**Visual hierarchy**: "Disbursed to school this term" (hero — eye lands here first) → active/completed/at-risk counts (secondary strip) → at-risk callout (if applicable) → arrangement list
**Key regions**:
- **Page header**: "Fee Top-Up Credit" + current term label
- **Hero metric**: DisbursedMetricHero — full-width or prominently sized card for "Disbursed to school this term". Visually distinct from the secondary metrics below it — larger type, stronger card treatment.
- **Secondary metric strip**: 3 equal MetricCards below the hero — Active, Completed, At risk
- **At-risk callout** (conditional): Shown only when AT_RISK count > 0
- **Read-only notice**: Subtle persistent indicator that this dashboard is observation-only. Prevents bursar confusion when parents call expecting action.
- **Escalation route**: Persistent contact link for Zeraki Finance support — bursar's relay path for parent enquiries they cannot resolve
- **Arrangement list**: DataTable of all arrangements this term

## Screen 8: Individual Arrangement Detail

**Purpose**: Give the bursar full detail on one credit arrangement — disbursement timeline, waterfall allocation history, and status — so they can answer parent enquiries without escalating.
**Actor**: Bursar
**Triggered by**: Click an arrangement row on Screen 7
**Journey reference**: J9 — Step 2, J10 (disbursement failed state)

---

### Layout Intent

**Structure**: Full-width within Finance shell. Page header with student + parent context and arrangement status. Two-column: left (wider) — disbursement timeline + upcoming repayments + waterfall allocation history. Right (narrower) — arrangement metadata + escalation route.
**Visual hierarchy**: Student name + arrangement status → disbursement amount and date (was the credit received?) → upcoming repayments (forward-looking — answers parent calls) → waterfall history (backward-looking) → escalation route
**Key regions**:
- **Arrangement header**: Student name, parent name, StatusBadge, credit amount, disbursement date
- **Disbursement timeline**: When and how much was credited to school
- **Upcoming repayments**: Forward-looking schedule of remaining instalments — date, amount due, status. Answers "how much do I still owe and when is my next payment?"
- **Waterfall allocation history**: Chronological list of past fee payments and how each was split
- **Metadata panel**: Consent channel, consent date, credit tier, arrangement ID
- **Escalation route**: Persistent Zeraki support contact — same as Screen 7

## Open UI Questions

- [ ] **OQ-UI-01**: Should the OfferBanner on Screen 2 be dismissible per session? If dismissible, it must re-appear on the next app open so the parent doesn't miss the offer. Define dismiss behaviour before design.
- [ ] **OQ-UI-02**: Should Consent Step 1 (Screen 4) include a "What does this mean?" expandable explanation for CRB reporting? The CRBFramingBlock is the primary mitigation — if it proves insufficient in user testing, the expandable is the next lever.
- [ ] **OQ-UI-03**: Legal review dependency — the exact consent copy on Screen 4 must be reviewed against CBK Digital Credit Provider Regulations 2022 and Kenya Data Protection Act 2019. The design must accommodate longer legal copy without truncating. Confirm copy length before finalising Screen 4 layout.
- [ ] **OQ-UI-04**: Should the Credit Dashboard (Screen 7) show a notification badge or alert when a new disbursement occurs or a DISBURSEMENT_FAILED event fires? The journey recommends this (J9, J10) but the feature spec does not require it. Product decision needed.
- [ ] **OQ-UI-05**: Screen 6 (Post-Acceptance) transitions between processing / disbursed / failed states. How does this state update reach the parent? Push notification → screen refresh? Polling? WebSocket? This is a technical architecture question that affects the UX — the "Fee top-up processing…" state may be the only state the parent sees if they close the app before disbursement. Flag for technical spec.
- [ ] **OQ-UI-06**: J4 flags a potential grace period before DECLINED is finalised (24-hour reversal window). If this is added, Screen 3 "No thanks" needs a confirmation step and Screen 2 needs a "reconsider" state. Defer until product decision on grace period.

## Upstream Feedback → journeys/1.0.2-zeraki-fee-top-up-credit.md

- **J1 — Layer 1 dependency gate copy confirmed**: The journey's recommended copy for the dependency gate ("Fee Top-Up Credit requires Instalment Plans…") is implemented verbatim on Screen 1. The CTA links directly to Layer 1 settings as the journey specifies. No changes needed.
- **J2 — WaterfallExpectationCallout**: The journey correctly identifies acceptance as the primary leverage point for J6 confusion. This spec implements the callout on both Screen 3 (offer screen) and Screen 6 (post-acceptance) — double exposure is intentional given the consent flow between them. Confirm with the journey designer that this is the right level of repetition.
- **J3 — FEES keyword**: The journey's recommendation to change the keyword from TOPUP to FEES (to avoid airtime/telco confusion) should be confirmed in the feature spec as a formal change. The compliance rationale (TOPUP is a fraud-adjacent keyword in the Kenyan SMS context) must be recorded in the feature spec, not only in the journey.
- **J5 — Expiry SMS copy**: The journey recommends changing the expiry SMS from "Contact the school if you still need support" to "Eligible parents will be reassessed at the start of next term." This is a system SMS copy requirement. It must be added to `conventions/micro-copy.md` and confirmed in the technical spec. The UI spec cannot enforce it.
- **J8 — AT_RISK bridging SMS**: The journey recommends adding a bridging SMS to the parent at the AT_RISK transition (currently absent from the feature spec). The journey describes this as a new step in Flow 10 and a new SMS Contracts entry. This spec surfaces the AT_RISK state in the bursar dashboard (Screen 7, Screen 8) but cannot enforce the SMS — it must be added to the feature spec before the technical spec is written.
