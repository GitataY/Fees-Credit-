## Screen 1: Settings — Instalment Plans

**Purpose**: Allow the Headteacher / School Admin to enable instalment plans, accept the school commitment, configure plan templates, and set the approval mode.
**Actor**: Headteacher / School Admin
**Triggered by**: Navigate to Settings → Fee Collection → Instalment Plans
**Journey reference**: J1 — School Enables Instalment Plans

---

### Layout Intent

**Structure**: Full-width settings page within the Zeraki Finance shell. Content in a single centered column, max-width constrained (not edge-to-edge). Existing Finance module sidebar and header persist.
**Visual hierarchy**: Feature description headline first → commitment statement (prominent, card-contained) → Enable button. Post-enable: template builder section comes into focus directly below the commitment card.
**Key regions**:
- **Page header**: "Instalment Plans" title + breadcrumb (Settings → Fee Collection → Instalment Plans)
- **Feature description region**: Brief explanation of what the feature does for the school
- **Commitment card**: The opt-in commitment statement — must be visually distinct (elevated card, not inline text) so it cannot be scrolled past without reading
- **Template builder region**: Visible after feature is enabled; prompts creation of 1–3 templates
- **Approval mode region**: Visible after at least one template exists; radio group with two options
- **Save / action bar**: Save Settings button — sticky at the bottom or inline at the end of the form

---

### Component Inventory

#### PageHeader — Navigation
**Purpose**: Orient the admin within the Finance settings hierarchy
**Copy**:
- Breadcrumb: "Settings / Fee Collection / Instalment Plans"
- Page title: "Instalment Plans"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible | Title in heading scale, breadcrumb in muted secondary text |

---

#### FeatureDescriptionBlock — Content Block
**Purpose**: Explain what instalment plans do, framed around parent dignity and school control
**Copy**:
- Headline: "Give parents a structured way to pay"
- Body: "Instalment plans let parents request a payment schedule for their child's fees — by app or SMS. Bursars review and approve requests, or you can set plans to auto-approve. You stay in control; parents get dignity."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible in disabled state | Neutral background, standard body text |

---

#### CommitmentCard — Container / Callout
**Purpose**: Surface the school's opt-in commitment so the admin cannot overlook it before enabling
**Copy**:
- Label: "School commitment"
- Body: "By enabling instalment plans, this school agrees that any student with an active plan that is being honoured will not be sent home for fee arrears during the plan period."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Visible before enabling | Elevated card with a left border accent (warning/info tone — not red, not green). Text is full-size, not small print. |
| acknowledged | After admin clicks Enable | Card remains visible but visually softens (border muted) — commitment is now a record, not a gate |

> **Micro-copy note (J1 friction)**: The commitment card must have enough vertical space that the full text is visible without scrolling when the card is in view. Do not truncate or collapse it. Consider adding a "Print / Save" affordance to allow the admin to share with their board before enabling — flag as open UI question.

---

#### EnableInstalmentPlansButton — PrimaryButton
**Purpose**: Opt the school into the feature after reading the commitment
**Copy**: "Enable Instalment Plans"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Feature is disabled | Full-emphasis primary button, positioned below CommitmentCard |
| loading | Clicked; system is activating | Button shows spinner, label changes to "Enabling…", disabled |
| hidden | Feature is already enabled | Button is not rendered — replaced by template builder and approval mode sections |

---

#### TemplateBuilderSection — Section Container
**Purpose**: Guide the admin to create 1–3 plan templates before the feature is live for parents
**Copy**:
- Section headline: "Plan templates"
- Helper text: "Create the plan options parents will see when they request a payment plan. You can add up to 3 templates per term."
- Calculation note (inline, adjacent to template form): "Amounts are calculated at the time a parent requests a plan, based on their child's outstanding balance. Templates define the structure — not a fixed amount."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| locked | Feature not yet enabled | Section is not visible |
| empty | Feature enabled, no templates created | Section visible with helper text + Add Template button; no template rows |
| partial | 1–2 templates created | Template rows rendered + Add Template button enabled |
| full | 3 templates created | Template rows rendered; AddTemplateButton disabled with message: "You have reached the maximum of 3 templates per term." |

---

#### TemplateRow — Form Row (repeatable, max 3)
**Purpose**: Allow the admin to define a single plan template — name and schedule structure
**Copy**:
- Name field label: "Template name"
- Name field placeholder: "e.g. Weekly: 4 payments"
- Interval field label: "Payment interval"
- Interval options: "Weekly", "Fortnightly", "Monthly", "Custom"
- Instalment count label: "Number of instalments"
- Remove action label: "Remove"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Row added, fields empty | Empty inputs, Remove button active |
| filled | All fields have values | Inputs filled, no error styling |
| error | Required field missing on save attempt | Inline error below the empty field; row highlighted with error border |
| disabled | 3 templates already exist (row is the 3rd being added) | Not applicable — rows are only added when the Add button is clicked |

---

#### AddTemplateButton — SecondaryButton
**Purpose**: Add a new template row to the builder (up to 3)
**Copy**: "Add template"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Fewer than 3 templates exist | Enabled, secondary style |
| disabled | 3 templates exist | Disabled; InlineMessage shown below: "You have reached the maximum of 3 templates per term." |

---

#### ApprovalModeSection — Section Container
**Purpose**: Allow the admin to choose between manual and auto-approve for parent plan requests
**Copy**:
- Section headline: "Approval mode"
- Helper text: "Choose how parent plan requests are handled."
- Option 1 label: "Manual approval (recommended)"
- Option 1 description: "All parent requests go to the bursar's approval queue. Bursars can approve, adjust, or decline."
- Option 2 label: "Auto-approve"
- Option 2 description: "Parent requests that match a template are automatically approved without bursar review."
- Mode-change note: "You can change this at any time."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| locked | No templates created yet | Section not visible |
| default | At least one template exists | Visible; Manual approval pre-selected |
| selected | Admin has made a choice | Chosen option highlighted in radio group |

---

#### ApprovalModeRadioGroup — RadioGroup
**Purpose**: Toggle between manual and auto-approve modes
**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| manual-selected | Default; manual approval chosen | "Manual approval" radio checked |
| auto-selected | Admin selects auto-approve | "Auto-approve" radio checked |

---

#### SaveSettingsButton — PrimaryButton
**Purpose**: Save template and approval mode configuration; make the feature live
**Copy**: "Save settings"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| disabled | No templates created | Disabled; does not block the Enable flow, only the template-save flow |
| default | At least one template created | Enabled |
| loading | Saving in progress | Spinner, "Saving…", disabled |
| success | Settings saved | AlertBanner shown: "Instalment plans are now enabled. Parents can request plans via the Zeraki app or by SMS. Bursars can create plans directly from the student fee screen." |

---

#### InlineValidationMessage — Feedback (inline)
**Purpose**: Surface field-level errors without requiring a full form submit
**Copy** (error cases):
- Missing template name: "Please enter a template name."
- Missing interval: "Please select a payment interval."
- Missing instalment count: "Please enter the number of instalments."
- No templates on save attempt: "Add at least one plan template before enabling instalment plans. Templates let parents request a plan without calling the school."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| hidden | No error | Not rendered |
| visible | Validation triggered | Error text below the relevant field; field border turns error color |

---

#### AlertBanner — Feedback (page-level)
**Purpose**: Confirm successful save with orientation message for the admin
**Copy** (success): "Instalment plans are now enabled. Parents can request plans via the Zeraki app or by SMS. Bursars can create plans directly from the student fee screen."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| success | Settings saved | Green/positive tone banner at top of content area; dismissible |
| error | Save failed (system error) | Error tone banner; message: "Settings could not be saved. Please try again." |

---

### Screen States

| State | Trigger | What changes on screen |
|-------|---------|----------------------|
| disabled | First visit, feature off | FeatureDescriptionBlock + CommitmentCard + EnableInstalmentPlansButton visible; template and approval sections hidden |
| enabling | Admin clicks Enable | Button goes to loading state |
| template-setup | Enable completes | EnableButton hidden; TemplateBuilderSection and ApprovalModeSection appear; AlertBanner not yet shown; SaveSettingsButton disabled until template added |
| active | Settings saved | AlertBanner shown; page reflects current template list and approval mode; admin can edit |

---

### Transitions
**On Enable Instalment Plans**: Page transitions to `template-setup` state — template builder becomes the focus
**On Save Settings**: AlertBanner shown; page settles into `active` state
**On Remove Template**: Template row removed; if count drops below 3, AddTemplateButton re-enables
**On error (no templates)**: InlineValidationMessage shown below TemplateBuilderSection; SaveSettingsButton remains active so admin can try again after adding a template

## Screen 2: Fee Screen (Parent App)

> **OQ-05 dependency**: This screen exists only if the Zeraki parent-facing app is confirmed. If the app is a new build, this screen is a separate scope item. If OQ-05 resolves to "app does not exist", Screens 2–5 are deferred and Path A (J2) is not specced.

**Purpose**: Show the parent their child's current term fee balance and surface the option to request a payment plan.
**Actor**: Parent
**Triggered by**: Parent opens the Zeraki app and navigates to a child's fee record
**Journey reference**: J2 — Step 1

---

### Layout Intent

**Structure**: Mobile-first, single-column scroll. Top section: student identity (name, school, term). Mid section: fee balance summary. Below balance: primary CTA for payment plan. Bottom: payment history list.
**Visual hierarchy**: Student name → Outstanding balance (large, prominent figure) → Request a Payment Plan CTA → payment history
**Key regions**:
- **Student header**: Name, school, current term
- **Balance card**: Outstanding fee amount — the dominant visual element on the screen
- **Plan CTA region**: "Request a Payment Plan" — positioned directly below the balance card, not in a warning or crisis state. Reads as a helpful option.
- **Payment history**: Below the fold; secondary information

---

### Component Inventory

#### BalanceCard — Data Display
**Purpose**: Surface the outstanding fee balance clearly so the parent understands what they owe
**Copy**:
- Label: "Outstanding balance"
- Amount: KES [amount] (large numeral)
- Term label: "[Term name], [Year]"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Data fetching | Skeleton screen — BalanceCard placeholder shows two skeleton lines |
| populated | Balance loaded | Amount in large type; positive balance in neutral (not red — fee arrears are normal, not alarming) |
| zero | No outstanding balance | Amount shows KES 0; Request Plan CTA hidden |

---

#### RequestPlanButton — PrimaryButton
**Purpose**: Entry point to the plan request flow
**Copy**: "Request a payment plan"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Balance exists, no active plan, templates configured | Enabled; positioned below BalanceCard |
| hidden | Balance is zero | Not rendered |
| disabled | Active plan already exists this term | Not rendered — replaced by PlanStatusBadge (see below) |
| unavailable | No templates configured for this term | Button replaced by InlineMessage: "Payment plans are not available yet this term. Please contact the school directly." |

---

#### PlanStatusBadge — StatusBadge
**Purpose**: Show the parent the current plan status when a plan is already in flight
**Copy**:
- PENDING_APPROVAL: "Plan request pending review"
- ACTIVE: "Payment plan active"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| pending | Plan request awaiting bursar review | Neutral/info badge; tappable — links to plan detail |
| active | Plan approved and running | Positive/green badge; tappable — links to plan detail |

---

### Screen States

| State | Trigger | What changes on screen |
|-------|---------|----------------------|
| loading | Screen opens | BalanceCard and list show skeleton |
| populated-no-plan | Data loads, no active plan | RequestPlanButton visible |
| populated-pending | Data loads, plan pending approval | PlanStatusBadge (pending) replaces RequestPlanButton |
| populated-active | Data loads, active plan running | PlanStatusBadge (active) replaces RequestPlanButton |
| error | Network / data error | AlertBanner: "Could not load fee details. Please check your connection and try again." |

---

### Transitions
**On tap Request a payment plan**: Navigate to Screen 3 — Template Selection
**On tap PlanStatusBadge**: Navigate to plan detail view (out of scope for this spec — parent-facing plan detail is not covered in the journey)

## Screen 3: Template Selection (Parent App)

**Purpose**: Let the parent choose a payment plan structure from the school's configured templates.
**Actor**: Parent
**Triggered by**: Tap "Request a payment plan" on Screen 2
**Journey reference**: J2 — Step 2

---

### Layout Intent

**Structure**: Mobile-first, single-column. Top: screen title + outstanding balance reminder. Middle: template option list (up to 3 items). Bottom: Cancel action.
**Visual hierarchy**: Screen title → balance reminder (anchors the choice to a real number) → template options (selectable cards) → Cancel
**Key regions**:
- **Screen header**: Back arrow + "Choose a plan" title
- **Balance reminder**: Small, muted — "Outstanding: KES [amount]". Keeps the parent grounded without making this feel like a crisis screen.
- **Template list**: Up to 3 TemplateOptionCards — one per template. Each shows the plan name and structure. The calculation note appears below the list.
- **Calculation note**: "Amounts are calculated based on [Student]'s outstanding balance. You'll see the exact schedule on the next screen."
- **Cancel link**: Low-emphasis; "Cancel" text link or SecondaryButton

---

### Component Inventory

#### TemplateOptionCard — SelectableCard
**Purpose**: Present a single plan template as a tappable option
**Copy**:
- Template name: "[Template name]" (e.g. "Weekly: 4 payments")
- Structure label: "[N] payments, [interval]"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Not selected | Card with border, standard weight |
| selected | Tapped by parent | Border highlights; checkmark or radio indicator active |
| loading | Schedule calculation triggered on tap | Brief loading indicator before navigating to Screen 4 |

> **Friction note (J2)**: Template names alone ("Weekly: 4 payments") don't tell the parent the KES amounts. The calculation note below the list is required to set correct expectations. If designs show space for a preview amount per card, flag as enhancement — the journey does not spec this for MVP.

---

#### CalculationNoteText — Body Text / Helper
**Purpose**: Prevent the parent from thinking they are choosing a fixed-amount plan
**Copy**: "Amounts are calculated based on [Student]'s outstanding balance. You'll see the exact schedule on the next screen."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always visible when templates are shown | Muted helper text below template list |

---

#### CancelLink — SecondaryButton / Text Link
**Purpose**: Allow the parent to exit the flow without submitting a request
**Copy**: "Cancel"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| default | Always available | Low-emphasis; does not compete with template options |

---

### Screen States

| State | Trigger | What changes on screen |
|-------|---------|----------------------|
| loading | Screen opens | Template list shows skeleton cards |
| populated | Templates loaded | Up to 3 TemplateOptionCards visible |
| error | Templates failed to load | AlertBanner: "Could not load plan options. Please try again." |

---

### Transitions
**On tap template card**: Navigate to Screen 4 — Schedule Preview with selected template
**On tap Cancel**: Return to Screen 2 — Fee Screen; no request created

## Screen 4: Schedule Preview (Parent App)

---

- **Instalment schedule list**: One InstalmentScheduleRow per instalment — date and amount

## Screen 6: Instalment Plans Dashboard

**Purpose**: Give the bursar a live overview of fee collection via plans — active plans, adherence, pending queue, and upcoming/overdue instalments.
**Actor**: Bursar
**Triggered by**: Navigate to Instalment Plans section in Zeraki Finance sidebar
**Journey reference**: J11 — Bursar Reviews the Instalment Plan Dashboard

---

### Layout Intent

**Structure**: Full-width web layout within Zeraki Finance shell. Sidebar persists. Content area: top row of MetricCards, then two-column middle zone (queue alert left, upcoming due right), then full-width plan list.
**Visual hierarchy**: Term-level metric summary first (anomalies pop here) → pending queue alert (if items exist) → upcoming obligations → full plan list
**Key regions**:
- **Page header**: "Instalment Plans" + current term label + "New Plan" button (links to Screen 12 via student search)
- **Metric row**: 4–5 MetricCards — at-a-glance term health
- **Alert strip**: Pending queue count (if > 0) — clickable, takes bursar to Screen 8
- **Upcoming / Overdue panels**: Two-column: "Due in 7 days" list left, "Overdue" list right
- **Plan list**: Full-width DataTable of all plans this term — filterable by status

---

### Component Inventory

#### MetricCard — Data Display (×5)
**Purpose**: Surface term-level headline figures for at-a-glance monitoring
**Cards and copy**:
1. "Active plans" — count
2. "KES expected this term" — sum of all active plan totals
3. "KES received" — sum paid to date
4. "On track" — count of plans with no missed instalments
5. "Overdue instalments" — count of MISSED instalments across all plans

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Data fetching | Skeleton tile |
| populated | Data loaded | Metric value in large type, label below in muted text |
| zero | No data yet this term | Value shows 0; no special error state |
| alert | Overdue count > 0 | "Overdue instalments" card uses alert/warning color treatment to draw the eye |

---

#### PendingQueueAlertStrip — AlertBanner / Callout
**Purpose**: Surface pending requests that need bursar attention without being buried in the plan list
**Copy**:
- "[N] plan request(s) waiting for your review"
- CTA: "Review queue"

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| hidden | No pending requests | Not rendered |
| visible | ≥1 pending request | Info/action tone banner; "Review queue" link navigates to Screen 8 |

---

#### UpcomingInstalmentsList — Data List
**Purpose**: Show instalments due within the next 7 days so the bursar can anticipate incoming cash and flag families who haven't paid
**Copy**:
- Section header: "Due in the next 7 days"
- Row format: "[Student name] — KES [amount] due [Day, Date]"
- Empty state: "No instalments due in the next 7 days."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Fetching | Skeleton rows |
| populated | Items exist | Rows listed; tappable — each row links to Screen 7 (Plan Detail) |
| empty | No upcoming | EmptyState component with copy above |

---

#### OverdueInstalmentsList — Data List
**Purpose**: Surface missed instalments that need follow-up
**Copy**:
- Section header: "Overdue"
- Row format: "[Student name] — KES [amount] was due [Day, Date]"
- Empty state: "No overdue instalments."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Fetching | Skeleton rows |
| populated | Items exist | Rows listed in alert/warning tone; tappable — links to Screen 7 |
| empty | None overdue | EmptyState with copy above |

---

#### PlanListTable — DataTable
**Purpose**: Full list of all plans this term — filterable by status — for detailed review and navigation
**Copy**:
- Table header: Student, Parent, Template, Status, Adherence, Last activity
- Status filter options: "All", "Active", "Pending approval", "Completed", "Cancelled", "Missed"
- Empty state: "No instalment plans this term. Plans appear here when parents request them or bursars create them."

**States**:
| State | Description | Visual treatment |
|-------|-------------|-----------------|
| loading | Data fetching | Skeleton rows |
| populated | Plans exist | Rows with StatusBadge per plan; rows tappable → Screen 7 |
| filtered-empty | Filter returns no results | "No plans match this filter." with clear-filter link |
| empty | No plans at all | EmptyState with copy above |

---

#### StatusBadge — Data Display (inline)
**Purpose**: Show plan or instalment status at a glance in lists and tables
**Values and visual treatment**:
| Value | Copy | Color treatment |
|-------|------|----------------|
| PENDING_APPROVAL | "Pending review" | Neutral / info |
| ACTIVE | "Active" | Positive / green |

|-------|---------|----------------------|
| default | Screen opens | Form empty (date pre-filled with today) |
| filled | Fields completed | Form valid; RecordPaymentSubmitButton enabled |
| checking | Submit clicked | Button loading |
| duplicate-detected | System finds possible duplicate | DuplicatePaymentDialog opens as modal |
| applied | Payment applied (no duplicate, or duplicate resolved) | ToastNotification: "Payment of KES [amount] recorded for [Student name]."; navigate back to Screen 7 |

---

### Transitions
**On Record payment (no duplicate)**: Apply payment; show toast; navigate to Screen 7
**On Record payment (duplicate detected)**: Open DuplicatePaymentDialog
**On Dismiss this entry**: Close dialog; original payment retained; navigate to Screen 7
**On Apply both**: Close dialog; both payments applied; navigate to Screen 7; audit note appended to plan history

## Upstream Feedback → journeys/1.0.1-zeraki-instalment-plans.md

- **J2 & J3 — Adjusted plan SMS copy**: The journey correctly identifies the "small adjustment" SMS as problematic and proposes "To give you more time…" framing. That framing has been revised — a bursar may adjust in either direction (more or fewer instalments), so directional framing is incorrect. The confirmed copy is: "Your plan has been adjusted to [N] payments of KES [amount]. Next due: [date]. Pay to [paybill] as usual." This should replace the journey's recommended copy, be added to `conventions/micro-copy.md`, and confirmed as a system-level requirement in the technical spec.
- **J5 — Decline SMS dead end**: The journey correctly identifies the decline SMS as requiring a concrete next step. The DeclineReasonForm in Screen 9 implements this via a required "next step" field. Confirm with the journey that this is the right resolution — the bursar enters the parent-facing next step at decline time.
- **J6 — Reminder SMS partial payment**: The journey correctly flags that the reminder SMS must use remaining balance, not original amount. This is a technical requirement for the reminder job. The UI spec cannot enforce this — it must be in the technical spec.
- **J8 — Backdating**: The journey notes audit risk for backdating. Screen 14 implements an InlineBackdateNotice when the entered date differs from today. Confirm this is the right friction level — the journey suggests "consider" but doesn't mandate it.
