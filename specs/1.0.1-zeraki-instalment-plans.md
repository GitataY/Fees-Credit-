# Feature 1 — School Setup: Enabling Instalment Plans

Feature: School enables instalment plans
  As a headteacher or school admin
  I want to opt my school into instalment plans and configure plan templates
  So that parents can request structured payment plans and bursars can manage fee collection

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: Admin enables instalment plans after creating at least one template
    Given the admin is authenticated in Zeraki Finance with settings permissions
    And no instalment plan templates exist for the current term
    When the admin creates a plan template named "Weekly: 4 payments" with 4 instalments on a weekly interval
    And the admin clicks Enable Instalment Plans and confirms the school commitment
    Then instalment plans are active for the school for the current term
    And the admin sees: "Instalment plans are now enabled. Parents can request plans via the Zeraki app or by SMS. Bursars can create plans directly from the student fee screen."

  @phase1 @instalment-plans @happy-path
  Scenario: Admin enables instalment plans with auto-approve mode selected
    Given the admin has created at least one plan template for the current term
    When the admin selects "Auto-approve" as the approval mode and enables instalment plans
    Then parent plan requests matching a template are automatically approved without bursar review
    And instalment plans are active for the school

  @phase1 @instalment-plans @happy-path
  Scenario: Admin creates up to three plan templates for the current term
    Given instalment plans are enabled for the school
    And no plan templates exist for the current term
    When the admin creates three plan templates for the current term
    Then all three templates are available for parents to choose from when requesting a plan

  # ---- Edge Cases ----

  @phase1 @instalment-plans @edge-case
  Scenario: Admin deactivates a template that has not been used by any plan
    Given the school has two active plan templates for the current term
    And neither template has been used by any plan request
    When the admin deactivates one template
    Then the deactivated template is no longer shown to parents as an option
    And the remaining active template is still available

  # ---- Error Paths ----

  @phase1 @instalment-plans @error-path
  Scenario: Admin tries to enable instalment plans without creating a template first
    Given the admin is on the Instalment Plans settings screen
    And no plan templates have been created for the current term
    When the admin clicks Enable Instalment Plans
    Then the system blocks activation
    And the admin sees: "Add at least one plan template before enabling instalment plans. Templates let parents request a plan without calling the school."

  @phase1 @instalment-plans @error-path
  Scenario: Admin tries to create a fourth plan template when three already exist
    Given the school has three active plan templates for the current term
    When the admin attempts to add another template
    Then the Add Template button is disabled
    And the admin sees: "You have reached the maximum of 3 templates per term."

# Feature 2 — Parent Requests Plan via App (Path A)

Feature: Parent requests a payment plan via the Zeraki app
  As a parent
  I want to request a structured instalment plan for my child's fees
  So that I can manage payment flexibly without gate negotiation

  Background:
    Given the school has instalment plans enabled with manual approval mode
    And at least one plan template is configured for the current term
    And the parent is authenticated in the Zeraki app
    And the student has an outstanding fee balance for the current term
    And no active instalment plan exists for the student this term

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: Parent requests a plan that enters the approval queue
    Given the parent is on their child's current term fee screen
    When the parent selects "Request a Payment Plan"
    And the parent selects a plan template
    And the parent reviews the calculated instalment schedule and confirms
    Then a plan request is created in PENDING_APPROVAL status
    And the parent receives an SMS: "[School]: Your plan request for [Student] is being reviewed. You'll hear back shortly."
    And the plan request appears in the bursar's approval queue

  @phase1 @instalment-plans @happy-path
  Scenario: Parent requests a plan at a school with auto-approve enabled
    Given the school's approval mode is auto-approve
    When the parent selects a plan template and confirms the instalment schedule
    Then the plan is immediately created in ACTIVE status
    And the parent receives an SMS: "[School]: Payment plan set up for [Student]. [N] payments of KES [amount]. Next due: [date]. Pay to [paybill] as usual."
    And no approval queue entry is created for the bursar

  # ---- Error Paths ----

  @phase1 @instalment-plans @error-path
  Scenario: Parent tries to request a plan when no templates are configured for this term
    Given no plan templates have been configured for the current term
    When the parent selects "Request a Payment Plan"
    Then the parent sees: "Payment plans are not available yet this term. Please contact the school directly."
    And no plan request is created

  @phase1 @instalment-plans @error-path
  Scenario: Parent tries to request a plan when one is already active for the student this term
    Given an active instalment plan already exists for the student this term
    When the parent selects "Request a Payment Plan"
    Then the parent sees: "[Student name] already has an active payment plan this term. Contact the school if you need to make changes."
    And no new plan request is created

# Feature 3 — Parent Requests Plan via SMS (Path B)

Feature: Parent requests a payment plan via SMS
  As a parent
  I want to request a payment plan by sending an SMS keyword
  So that I can set up a structured plan without needing the app

  Background:
    Given the school has instalment plans enabled with manual approval mode
    And at least one plan template is configured for the current term
    And the parent's phone number is registered against the student in Zeraki Finance
    And the student has an outstanding fee balance for the current term
    And no active instalment plan exists for the student this term

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: Parent completes a plan request via SMS with a single student on file
    Given the parent's phone number is linked to exactly one student
    When the parent sends "PLAN" to the school's SMS number
    Then the system sends the available templates: "[School name] payment plans for [Student name]: Reply 1 for [Template 1 name] | Reply 2 for [Template 2 name]..."
    When the parent replies with a valid template number
    Then the system sends the calculated schedule: "Plan: [N] payments of KES [amount]. Due dates: [date 1], [date 2]... Reply YES to confirm or NO to cancel."
    When the parent replies "YES"
    Then a plan request is created in PENDING_APPROVAL status
    And the parent receives a confirmation SMS
    And the plan request appears in the bursar's approval queue

  @phase1 @instalment-plans @happy-path
  Scenario: Parent with multiple students linked to their number is prompted to specify a child
    Given the parent's phone number is linked to two students
    When the parent sends "PLAN" to the school's SMS number
    Then the system sends: "Reply with the student number: 1. [Student A] 2. [Student B]"
    When the parent replies with the number for one student
    Then the system continues the plan request flow for that student

  # ---- Edge Cases ----

  @phase1 @instalment-plans @edge-case
  Scenario: SMS session expires after 10 minutes without a valid reply
    Given the parent has received the template selection SMS
    When 10 minutes pass without the parent sending a valid reply
    Then the SMS session expires silently
    And no plan request is created
    And the parent can restart by sending "PLAN" again

  # ---- Error Paths ----

  @phase1 @instalment-plans @error-path
  Scenario: PLAN keyword sent from a phone number not registered in Zeraki
    Given the sending phone number is not registered to any student in Zeraki Finance
    When a parent sends "PLAN" to the school's SMS number
    Then the system replies: "We couldn't find your details. Please contact [School name] directly."
    And no plan request is created

  @phase1 @instalment-plans @error-path
  Scenario: Parent replies with an invalid option number during template selection
    Given the parent has received the template selection SMS showing 2 templates
    When the parent replies with "5"
    Then the system replies: "Please reply with 1, 2, or 3 to choose a plan, or contact the school directly."
    And the SMS session remains open awaiting a valid reply

  @phase1 @instalment-plans @error-path
  Scenario: Parent cancels at the confirmation step
    Given the parent has received the schedule confirmation SMS
    When the parent replies "NO"
    Then the system replies: "No plan has been set up. You can start over by sending PLAN again."
    And no plan request is created

# Feature 5 — Bursar Processes the Approval Queue

Feature: Bursar processes payment plan requests from the approval queue
  As a bursar
  I want to review, approve, adjust, or decline parent plan requests
  So that only appropriate plans are activated for students

  Background:
    Given the school has instalment plans enabled with manual approval mode
    And the bursar is authenticated in Zeraki Finance with fee management permissions
    And at least one plan request is in PENDING_APPROVAL status

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: Bursar approves a plan request as submitted
    When the bursar opens the approval queue and selects a pending request
    And the bursar clicks Approve
    Then the plan request transitions to APPROVED status
    And an ACTIVE plan is created matching the requested template and schedule
    And the parent receives an SMS: "[School]: Payment plan set up for [Student]. [N] payments of KES [amount]. Next due: [date]. Pay to [paybill] as usual."

  @phase1 @instalment-plans @happy-path
  Scenario: Bursar adjusts a plan request before approving it
    When the bursar opens a pending request and clicks Adjust
    And the bursar modifies the instalment amounts and due dates
    And the bursar saves and approves the adjusted plan
    Then the plan request transitions to ADJUSTED status
    And an ACTIVE plan is created with the bursar's adjusted schedule
    And the parent receives an SMS: "[School]: Your plan has been adjusted to [N] payments of KES [amount]. Next due: [date]. Pay to [paybill] as usual."

  @phase1 @instalment-plans @happy-path
  Scenario: Bursar declines a plan request with a reason and a next step
    When the bursar opens a pending request and clicks Decline
    And the bursar enters a decline reason and a next step for the parent
    Then the plan request transitions to DECLINED status
    And the parent receives an SMS: "[School]: Your plan request for [Student] could not be approved: [reason]. [next_step]."
    And no plan is created for the student

  @phase1 @instalment-plans @happy-path
  Scenario: Bursar declines a plan request without entering a reason
    When the bursar clicks Decline without entering a reason
    Then the plan request transitions to DECLINED status
    And the parent receives an SMS: "[School]: Your plan request for [Student] could not be approved. Please contact the school."
    And no plan is created for the student

  @phase1 @instalment-plans @happy-path
  Scenario: Bursar bulk-approves all pending requests using Approve All
    Given three plan requests are in PENDING_APPROVAL status
    When the bursar clicks Approve All
    Then all three requests transition to APPROVED status
    And an ACTIVE plan is created for each student
    And each parent receives a plan active SMS

# Feature 6 — Automated SMS Reminders

Feature: Automated SMS reminders sent before instalment due dates
  As a parent
  I want to receive a reminder SMS before my instalment is due
  So that I don't miss a payment

  Background:
    Given the school has instalment plans enabled
    And a student has an ACTIVE instalment plan with at least one PENDING instalment

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: System sends a reminder SMS within the configured window before a due date
    Given a PENDING instalment's due date falls within the configured reminder window
    And no reminder has already been sent for this instalment in this reminder cycle
    And the plan is still ACTIVE
    When the daily scheduled reminder job runs
    Then the parent receives an SMS: "[School]: Reminder — KES [amount] due for [Student] on [date]. Pay to [paybill] as usual."
    And the reminder is recorded against the instalment with a sent timestamp and SENT delivery status

  # ---- Edge Cases ----

  @phase1 @instalment-plans @edge-case
  Scenario: Reminder for a partially paid instalment shows only the outstanding amount
    Given a PENDING instalment with a full amount of KES 3,750
    And the parent has already paid KES 2,000 toward that instalment
    And the instalment's due date falls within the configured reminder window
    When the daily scheduled reminder job runs
    Then the parent receives an SMS showing KES 1,750 due — not KES 3,750
    And the reminder is recorded against the instalment

  @phase1 @instalment-plans @edge-case
  Scenario: Reminder is not re-sent for an instalment that already received one this cycle
    Given a reminder SMS has already been sent for a PENDING instalment in the current reminder cycle
    When the daily scheduled reminder job runs again
    Then no second reminder is sent for that instalment

  @phase1 @instalment-plans @edge-case
  Scenario: Reminder SMS delivery fails
    Given a PENDING instalment is within the reminder window
    When the daily reminder job runs but the SMS gateway returns a delivery failure
    Then the reminder delivery status is recorded as FAILED on the instalment record
    And the bursar's dashboard surfaces the delivery failure
    And the bursar can manually resend the reminder from the instalment detail screen

# Feature 7 — Payment Matched to Active Plan (M-Pesa)

Feature: Fee payment automatically matched to an active instalment plan
  As a parent
  I want my M-Pesa fee payments automatically applied to my instalment plan
  So that my plan progress is tracked without any extra steps

  Background:
    Given the school has instalment plans enabled
    And the student has an ACTIVE instalment plan for the current term
    And the student has at least one PENDING instalment

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: Full payment received on or before the due date marks instalment as paid on time
    Given the oldest PENDING instalment is due on a future date
    When a fee payment equal to the instalment amount arrives via the school's M-Pesa paybill
    Then the instalment is marked PAID_ON_TIME
    And the plan's on-time rate and completion rate are recalculated
    And the parent receives an SMS: "[School]: KES [amount] received for [Student]. Remaining plan balance: KES [amount]. Next due: KES [amount] on [date]."

  @phase1 @instalment-plans @happy-path
  Scenario: Payment of the final instalment completes the plan
    Given only one PENDING instalment remains on the plan
    When a payment equal to that instalment amount is received
    Then the final instalment is marked PAID_ON_TIME
    And the plan transitions to COMPLETED status
    And the parent receives an SMS: "[School]: [Student]'s payment plan is complete. All instalments paid. Thank you."

  # ---- Edge Cases ----

  @phase1 @instalment-plans @edge-case
  Scenario: Full payment received after the due date marks instalment as paid late
    Given the oldest PENDING instalment's due date has passed
    When a fee payment equal to the instalment amount arrives
    Then the instalment is marked PAID_LATE
    And the plan's completion rate is incremented
    And the plan's on-time rate is not incremented
    And the parent receives a payment received SMS

  @phase1 @instalment-plans @edge-case
  Scenario: Partial payment is recorded but does not change instalment status
    Given the oldest PENDING instalment amount is KES 2,000
    When a fee payment of KES 1,000 arrives
    Then the instalment remains in PENDING status
    And the paid amount on the instalment is updated to KES 1,000
    And the adherence metrics are not changed
    And the parent receives a payment received SMS

  @phase1 @instalment-plans @edge-case
  Scenario: Payment exceeding the total remaining plan amount completes the plan and credits the surplus
    Given the plan has a remaining balance of KES 3,000
    When a fee payment of KES 4,500 arrives via the school's M-Pesa paybill
    Then all remaining PENDING instalments are marked PAID_ON_TIME
    And the plan transitions to COMPLETED status
    And KES 1,500 is credited to the student's general fee balance
    And the parent receives a plan complete SMS

  @phase1 @instalment-plans @edge-case
  Scenario: M-Pesa payment arrives for a student with no active plan and is processed as a standard fee payment
    Given no active instalment plan exists for the student this term
    When a fee payment arrives via the school's M-Pesa paybill
    Then the payment is processed through standard Zeraki Finance receipting
    And no plan matching occurs
    And no plan-related SMS notification is sent

# Feature 8 — Bursar Records Manual Payment

Feature: Bursar records a manual cash or bank deposit payment against a plan
  As a bursar
  I want to record cash and bank deposit payments against a student's active plan
  So that parents who pay outside M-Pesa are correctly reflected in adherence records

  Background:
    Given the school has instalment plans enabled
    And the bursar is authenticated in Zeraki Finance with fee management permissions
    And the student has an ACTIVE instalment plan for the current term

  # ---- Happy Path ----

  @phase1 @instalment-plans @happy-path
  Scenario: Bursar records a cash payment that is applied to the oldest pending instalment
    Given no duplicate payment exists for the same student, amount, and date
    When the bursar records a cash payment with the amount, payment date, and payment method
    Then the payment is allocated to the oldest PENDING instalment using FIFO logic
    And on-time or late status is determined by the bursar-entered payment date relative to the instalment due date
    And both adherence metrics are recalculated
    And the parent receives a payment received SMS

  # ---- Edge Cases ----

  @phase1 @instalment-plans @edge-case
  Scenario: Bursar backdates a manual payment and on-time status reflects the entered date
    Given a student's instalment was due 5 days ago
    And the parent paid in cash 3 days ago but the payment was not yet entered
    When the bursar records the payment with a payment date set to 3 days ago
    Then the instalment is marked PAID_LATE based on the entered payment date
    And today's entry date is not used to determine on-time or late status

  @phase1 @instalment-plans @edge-case
  Scenario: Duplicate detected — bursar dismisses the manual entry as the same payment
    Given an M-Pesa payment of KES 2,000 was already recorded for the student on the same date
    When the bursar enters a KES 2,000 cash payment for the same student on the same date
    Then the system surfaces: "A M-Pesa payment of KES 2,000 was already recorded for [Student name] on [date]. Is this the same payment or a separate one?"
    When the bursar selects "Same payment — dismiss this entry"
    Then the manual entry is discarded
    And no allocation occurs against the plan

  @phase1 @instalment-plans @edge-case
  Scenario: Duplicate detected — bursar confirms both payments are separate and applies both
    Given an M-Pesa payment of KES 2,000 was already recorded for the student on the same date
    When the bursar enters a separate KES 2,000 cash payment for the same student on the same date
    And the system surfaces the duplicate warning
    And the bursar selects "Separate payment — apply both"
    Then both payments are applied independently to the plan in chronological order
    And the system logs the bursar's override decision

  @phase1 @instalment-plans @edge-case
  Scenario: M-Pesa webhook arrives matching an existing manual entry and is held for review
    Given the bursar has already recorded a KES 1,500 manual payment for a student on a given date
    When a KES 1,500 M-Pesa webhook payment arrives for the same student on the same date
    Then the webhook payment is held in PENDING_REVIEW status and is not applied to the plan
    And the bursar's dashboard shows: "[Student name]: a KES 1,500 M-Pesa payment may duplicate a manual entry from [date]. Review before applying."
    And neither payment affects the plan's adherence metrics or the parent's SMS notifications until the flag is resolved

  @phase1 @instalment-plans @edge-case
  Scenario: Bursar enters a payment amount that exceeds the remaining plan balance
    Given the plan's remaining balance is KES 2,000
    When the bursar records a payment of KES 3,500
    Then KES 2,000 is applied to complete the plan
    And KES 1,500 is credited to the student's general fee balance

  # ---- Error Paths ----

  @phase1 @instalment-plans @error-path
  Scenario: Manual payment recorded for a student with no active plan this term
    Given no active plan exists for the student this term
    When the bursar records a payment for the student
    Then the payment is receipted as a standard Zeraki Finance fee payment
    And no plan matching occurs and no plan-related SMS is sent

# Feature 12 — Bursar Cancels a Plan

Feature: Bursar cancels an active instalment plan
  As a bursar
  I want to cancel an active plan when it is no longer appropriate
  So that the student's record is accurate and the parent is informed

  Background:
    Given the school has instalment plans enabled
    And the bursar is authenticated in Zeraki Finance with fee management permissions
    And the student has an ACTIVE instalment plan with at least one paid instalment and one pending instalment

  # ---- Happy Path ----

  Scenario: Bursar cancels an active plan
    When the bursar opens the student's active plan and selects Cancel Plan
    And the bursar confirms the cancellation prompt
    Then the plan transitions to CANCELLED status
    And all remaining PENDING instalments are marked CANCELLED
    And the parent receives an SMS: "[School]: Your payment plan for [Student] has been cancelled. Please contact the school to discuss your fee balance."

  # ---- Edge Cases ----
